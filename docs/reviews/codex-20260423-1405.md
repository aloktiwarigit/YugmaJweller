The new settings screens can expose editable fallback state after an initial fetch failure. In both cases that creates a real path to overwrite existing server-side settings with defaults or empty values, so the patch is not safe to treat as correct.

- [P1] Block editing when making charges fail to load — C:\Alok\Business Projects\Goldsmith\apps\shopkeeper\app\settings\making-charges.tsx:25-35
  If `/api/v1/settings/making-charges` returns an error (offline, expired auth, 5xx), `isLoading` becomes `false` while `data` stays `undefined`, so the screen falls back to `DEFAULTS`. Pressing any save button in that state sends those defaults back to the API and can overwrite the tenant’s real making-charge configuration even though we never loaded it. This should stay in an error/retry state until the initial fetch succeeds.

- [P2] Don't render an empty profile form after the initial load fails — C:\Alok\Business Projects\Goldsmith\apps\shopkeeper\app\settings\shop-profile.tsx:43-59
  When `/api/v1/settings/profile` fails here, the component still clears `loading` and renders the form with all-empty local state. Several blur handlers then persist `null`/empty values (for example `bis_registration`, `contact_phone`, or `about_text`) even though the existing profile was never loaded, so a transient fetch failure can silently wipe fields the user touches. Keeping the form disabled behind an error/retry state would avoid that data loss path.

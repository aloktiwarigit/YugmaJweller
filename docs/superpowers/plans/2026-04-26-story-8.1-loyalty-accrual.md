# 8.1: Loyalty Points Auto-Accrual

> **For agentic workers:** Use `superpowers:executing-plans` (or subagent-driven). Steps use checkbox (`- [ ]`) syntax for tracking.

> **Class:** B (M complexity) — opens Epic 8 (Loyalty)
> **Codex weekly limit reached** → modified ceremony: TDD ✅ · feature-dev:code-reviewer ✅ · Codex ⏸ · git push ⏸
> **Epic:** E8 — Loyalty
> **PRD FRs:** FR-103 (auto-accrue points on invoice) · FR-104 (per-customer balance) · NFR (append-only loyalty ledger, parity with stock_movements pattern from 3.8)

**Goal:** When an invoice with a `customer_id` is issued, asynchronously credit the customer's loyalty points based on `shop_settings.loyalty_json.earnRatePercentage`, persist the credit in an append-only `loyalty_transactions` ledger, update the running aggregate in `customer_loyalty`, and expose read + manual-adjustment APIs.

**Architecture:**
- **Append-only ledger** (parity with `stock_movements` from 3.8): `loyalty_transactions` rejects UPDATE/DELETE via trigger; `app_user` is granted SELECT + INSERT only. Reversal/adjustment is a new row, never an edit.
- **Running aggregate** in `customer_loyalty` (one row per customer); FOR UPDATE serializes concurrent accruals.
- **Decoupled via domain event**: billing service emits `invoice.created`; LoyaltyModule listener enqueues a BullMQ job. Keeps invoice issuance fast and tolerant of loyalty failures.
- **Idempotent worker**: Redis key `loyalty:accrual:<invoiceId>` (TTL 7d) skips repeats on retry.
- **Walk-in safe**: invoices with `customer_id = null` skip loyalty entirely.
- **Tier transitions deferred** to Story 8.2; this story only accrues + records.

**Tech Stack:** PostgreSQL 15 RLS, NestJS, Drizzle, BullMQ, vitest, React Native (Expo).

## NEW Task 1.5 — Migration 0058: tenant-scoped composite FKs (resolves F1 + DB part of F7)

**Why:** Migration 0057 (already shipped at 0f65d0e) defines `product_images.product_id` and `product_images.uploaded_by_user_id` as plain FKs to `products(id)` and `shop_users(id)`. RLS catches `shop_id` filtering on read, but the FK constraint alone permits a row with `shop_id = X` referencing a `product_id = Y` that belongs to shop Z (≠ X). This is a real cross-tenant pollution / impersonation surface.

**Approach:** Don't amend 0057 (already verified against testcontainers; rewriting risks regression). Add a new migration 0058 that:

1. Adds `UNIQUE(shop_id, id)` on `products` and `shop_users` so a composite FK has a target.
2. Drops the existing plain FKs on `product_images`.
3. Adds composite tenant-scoped FKs.
4. Adds a `(product_id, idempotency_key)` UNIQUE constraint to support F7 (upload idempotency at DB layer — see Task 4 section below).

### SQL (drop into `packages/db/src/migrations/0058_product_images_tenant_fk.sql`)

```sql
-- packages/db/src/migrations/0058_product_images_tenant_fk.sql
-- Story 17.1 Codex round-1 follow-up — tenant-scoped composite FKs.
-- DDL-only per docs/db-workflow.md.
--
-- Why: 0057 used plain FKs (product_id -> products(id), uploaded_by_user_id ->
-- shop_users(id)). RLS filters on shop_id but does not enforce that the FK
-- target belongs to the same tenant. An attacker (compromised tenant or API
-- bug) could insert a product_images row pointing to another shop's product
-- or user. Composite FKs close this loophole at the schema layer.

BEGIN;

-- Step 1: Add UNIQUE(shop_id, id) on parent tables — required as composite FK
-- target. Both tables have shop_id NOT NULL today; pure metadata change.
ALTER TABLE products
  ADD CONSTRAINT products_shop_id_id_uniq UNIQUE (shop_id, id);

ALTER TABLE shop_users
  ADD CONSTRAINT shop_users_shop_id_id_uniq UNIQUE (shop_id, id);

-- Step 2: Drop the plain FKs from 0057.
ALTER TABLE product_images
  DROP CONSTRAINT product_images_product_id_fkey;

ALTER TABLE product_images
  DROP CONSTRAINT product_images_uploaded_by_user_id_fkey;

-- Step 3: Re-add as tenant-scoped composite FKs.
ALTER TABLE product_images
  ADD CONSTRAINT product_images_shop_product_fkey
  FOREIGN KEY (shop_id, product_id)
  REFERENCES products(shop_id, id)
  ON DELETE CASCADE;

ALTER TABLE product_images
  ADD CONSTRAINT product_images_shop_uploader_fkey
  FOREIGN KEY (shop_id, uploaded_by_user_id)
  REFERENCES shop_users(shop_id, id);

-- Step 4: F7 follow-on — idempotency key column + unique per-product constraint.
-- This supports the service-layer idempotency check in Task 4.
ALTER TABLE product_images
  ADD COLUMN idempotency_key TEXT;

CREATE UNIQUE INDEX product_images_idempotency_uniq
  ON product_images (product_id, idempotency_key)
  WHERE idempotency_key IS NOT NULL;

COMMIT;
```

## Task 4 — folded findings: F2, F3 (service), F7 (service), F8

### F2 (P2) — sharp error handling

**Plan lines:** 1322-1357
**Issue:** sharp `metadata()`, probe encoding, EXIF strip, and post-rotation metadata reads are not wrapped. Malformed image bytes that pass MIME-magic sniff can bubble as 500s.
**Fix:** Wrap the entire sharp pipeline in a single `try/catch`. Map any thrown error to a client-safe `BadRequestException` with code `INVALID_IMAGE` (or `INVALID_DIMENSIONS` if the failure is post-probe). Add a corrupt-JPEG fixture and red test:
```typescript
it('returns 400 INVALID_IMAGE for corrupt JPEG bytes that pass MIME sniff', async () => {
  const corrupt = Buffer.concat([Buffer.from([0xFF, 0xD8, 0xFF]), Buffer.alloc(1000, 0)]);
  await expect(svc.upload(...)).rejects.toMatchObject({ response: { code: 'INVALID_IMAGE' } });
});
```

### F3 (P2) — upload alt_text length validation

**Plan lines:** 1301-1392 (service), 1664-1672 (controller)
**Issue:** `upload()` accepts `alt_text` without the 200-char limit that `setAltText()` enforces (plan:1587-1589). Inconsistent validation lets the DB reject rows that the upload path accepted.
**Fix in service (Task 4):**
```typescript
if (input.altText !== null && input.altText !== undefined && input.altText.length > 200) {
  throw new BadRequestException({ code: 'ALT_TEXT_TOO_LONG' });
}
```
**Fix in controller (Task 6):** add `alt_text: z.string().max(200).nullable().optional()` to the upload Zod schema.
**Test:** red test in Task 4 — `expect(svc.upload({ altText: 'x'.repeat(201), ... })).rejects.toMatchObject({ response: { code: 'ALT_TEXT_TOO_LONG' } })`.

### F7 (P2) — upload idempotency

**Plan lines:** 1370-1404
**Issue:** Client timeout after DB commit + retry creates a second image with a new storage key, eats the 10-image cap, and orphans the first storage object.
**Fix:**
- DB layer (already covered by Task 1.5 — `(product_id, idempotency_key)` partial unique index)
- Service layer (Task 4):
  - Accept `idempotencyKey` (optional) on `upload()` input
  - Before insert: SELECT `product_images WHERE product_id = ? AND idempotency_key = ?`. If a row exists, return it (idempotent replay).
  - On INSERT, populate `idempotency_key`. Catch UNIQUE-violation `23505` on the partial index and re-fetch + return the existing row (handles the race where two concurrent retries both pass the SELECT).
- Controller layer (Task 6): accept `Idempotency-Key` HTTP header, pass through to service.

**Tests (Task 4):**
- `idempotent retry returns same image row, no second storage key`
- `concurrent retries: only one row created, both return same id`
- `different idempotency keys on same product: both create rows (independent uploads)`
- `null idempotency key: both create rows (legacy/no-replay path)`

## Task 5 — folded finding: F4 (service)

### F4 (P2) — reorder duplicate-IDs validation

**Plan lines:** 981-1001 (service `setSortOrders`), 1641 (controller schema)
**Issue:** `reorder()` checks list size matches but not uniqueness. With one image, `orderedIds = [id_1, id_1]` would match `currentImages.length === 1` and pass, then UPDATE the same row twice with conflicting sort_order values.
**Fix in service:**
```typescript
const unique = new Set(orderedIds);
if (unique.size !== orderedIds.length) {
  throw new BadRequestException({ code: 'ORDER_LIST_DUPLICATES' });
}
```
**Fix in controller (Task 6):** strengthen Zod schema:
```typescript
const reorderSchema = z.object({
  orderedIds: z.array(z.string().uuid()).min(1).refine(
    (ids) => new Set(ids).size === ids.length,
    { message: 'orderedIds must be unique' },
  ),
});
```
**Tests (Task 5):**
- `reorder rejects duplicate IDs in single-image case`
- `reorder rejects duplicate IDs in multi-image case`
- `reorder accepts unique IDs across the cap (10 entries)`

## Task 7 — folded finding: F6 (DTO)

### F6 (P2, server side) — server-built shopkeeper thumbnail URL

**Plan lines:** 2098-2103, 2179 (mobile hand-builds URL)
**Issue:** Spec says all ImageKit URLs go through the builder; mobile hand-builds them.
**Fix:** Two viable options. Pick whichever matches the broader codebase convention (check `packages/integrations/storage/src/image-kit-url-builder.ts` or wherever `imagekitUrl()` lives):

**Option A — server-built (simpler):** Extend the shopkeeper DTO to include a `thumbnail_url` field built server-side via the same `imagekitUrl()` builder used for the public DTO. Mobile renders `image.thumbnail_url` directly, no client-side URL construction.

**Option B — shared client package:** Move `ImageKitTransformUrlBuilder` (or just the `imagekitUrl()` pure function) into a `@goldsmith/storage-client` package. Mobile imports it and builds URLs client-side with the same code path.

**Recommendation: A.** Less surface, no new package, no risk of mobile getting a stale builder version. The server already has the URL builder; just expose the result.

### Action

In Task 7 (or Task 4 where the shopkeeper DTO is shaped), add `thumbnail_url: string` to `ImageRow`:

## Task 9 — folded findings: F5, F6 (mobile)

### F5 (P2) — mobile i18n key map

**Plan lines:** 2068-2073, 2141-2143
**Issue:** Mobile maps API error codes to i18n keys mechanically (`t('images_err_' + code.toLowerCase())`). For server codes like `PAYLOAD_TOO_LARGE` (HTTP 413 from Multer), `IMAGE_LIMIT_REACHED`, `IMAGE_TOO_LARGE_AFTER_COMPRESSION`, the i18n keys may not exist → falls back to generic `images_err_generic`.
**Fix:** Replace mechanical mapping with an explicit code-to-key map covering every server error code the upload/edit/delete flow can return:
```typescript
const ERROR_KEY_MAP: Record<string, string> = {
  INVALID_IMAGE:                    'images_err_invalid_image',
  INVALID_DIMENSIONS:               'images_err_invalid_dimensions',
  ALT_TEXT_TOO_LONG:                'images_err_alt_text_too_long',
  PAYLOAD_TOO_LARGE:                'images_err_payload_too_large',
  IMAGE_LIMIT_REACHED:              'images_err_limit_reached',
  IMAGE_TOO_LARGE_AFTER_COMPRESSION:'images_err_too_large_after_compression',
  ORDER_LIST_MISMATCH:              'images_err_order_mismatch',
  ORDER_LIST_DUPLICATES:            'images_err_order_duplicates',
  IMAGE_NOT_FOUND:                  'images_err_not_found',
};

function errorMessage(code: string | undefined): string {
  if (code && ERROR_KEY_MAP[code]) return t(ERROR_KEY_MAP[code]);
  return t('images_err_generic');
}
```

Add Hindi strings for each new key in the i18n bundle. Cover Multer's HTTP 413 (no JSON body) by inspecting `error.response.status === 413` and mapping to `PAYLOAD_TOO_LARGE`.

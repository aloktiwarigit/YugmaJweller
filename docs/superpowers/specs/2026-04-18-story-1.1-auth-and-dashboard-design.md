# Story 1.1 — Shop Owner reaches the empty shop dashboard (auth + dashboard on E2-S1 substrate)

## 1. Summary

Rajesh-ji, the anchor jeweller, opens a branded Expo app, enters his phone number, receives an OTP via Firebase, verifies it, and lands on his shop's empty-state dashboard in Hindi. Every subsequent API request he makes carries a verified Firebase ID token, resolves to his `shop_users` row, and enforces RLS against his `shop_id` — zero cross-tenant exposure, auth-audit-logged.

The story sits on top of the E2-S1 tenant-RLS substrate (shipped at commit `fc6af00`). It is the first story that:
- Introduces a real user identity (populates `TenantContext.userId` and `role`, previously `Partial<TenantContext>`).
- Exercises the tenant-isolation endpoint-walker against a real endpoint (previously only `/healthz`).
- Puts UI on the screen (first Expo app, first i18n strings, first Direction 5 tokens in running code).

**Scope discipline.** This story is intentionally a thin slice. It ships auth + a shell dashboard; it does NOT ship staff invite (1.2), role-based permissions (1.3), settings (Epic 2), products (Epic 3), invoices (Epic 5), customers (Epic 4), or any money primitives. `packages/money` is deferred to its first money-using story.

**Story size — honest flag.** Even after E2-S1 pulled the RLS + observability + CI substrate out, this story still introduces one new app (`apps/shopkeeper`), six new packages (`auth-client`, `ui-tokens`, `ui-mobile`, `i18n`, `secrets`), two migrations, a Firebase Admin integration, a Firebase RN client integration, four E2-S1 deferral closures, one new ADR, a PRD amendment note, and a Maestro e2e rig. This is XL. The plan-writing session SHOULD evaluate whether to execute it as a single PR or split into 1.1a (backend auth + migrations + SECURITY DEFINER + rate-limit + E2-S1 deferrals + ADR-0016) and 1.1b (shopkeeper app shell + Direction 5 + i18n + Maestro). Both paths are acceptable; the spec below is single-PR-shaped but factors cleanly into 1.1a + 1.1b along the `apps/` ↔ `packages/` seam if the plan session prefers a split. The user-facing story text ("Rajesh-ji reaches the dashboard") is inherently end-to-end and requires both halves to land before Story 1.2 can start.

### 4.1 Happy-path sequence (first-ever login for anchor owner)

1. App boot. `TenantProvider` reads `EXPO_PUBLIC_TENANT_SLUG` from `app.config.ts` (dev: `anchor-dev`). Calls `GET /api/v1/tenant/boot?slug=anchor-dev` — this endpoint is `@SkipTenant`, calls SECURITY DEFINER `tenant_boot_lookup(slug)` which returns `(id, display_name, config_jsonb)` for ACTIVE shops only. Response applies brand (logo URL, primary_color override, app_name) to the React tree. Direction 5 tokens are the baseline; `shops.config` overrides are additive.
2. `AuthProvider` subscribes to `firebase.auth().onAuthStateChanged`. Current state: unauthenticated → render `(auth)/phone.tsx`.
3. Owner taps phone input, types `9876543210`. App prepends `+91`, calls `auth().signInWithPhoneNumber('+919876543210')`. Firebase sends OTP via SMS (free Spark tier, anchor is on pre-verified test phones during dev).
4. SDK returns a `confirmation` handle. App navigates to `(auth)/otp.tsx`, starts a 60-second "पुनः भेजें" resend countdown, shows six-digit Devanagari-numeral input.
5. Owner types `123456`. App calls `confirmation.confirm('123456')`. On success, Firebase SDK persists the refresh token to native Keystore and returns a Firebase ID token (~1hr TTL).
6. App calls `getIdToken()` → POST `/api/v1/auth/session` with `Authorization: Bearer <firebase-id-token>`.
7. Server flow (`/auth/session` is `@SkipTenant` — no tenant context yet):
   a. `FirebaseJwtStrategy.validate(token)` → Admin SDK `verifyIdToken(token, true)` → returns `{uid, phone_number, ...}`.
   b. `AuthRateLimitService.checkVerify(phone_number)` — read `auth_rate_limits`. If `locked_until > now()` → 429 `auth.locked`. If `verify_failures >= 10` in rolling window → set `locked_until = now() + 15min`, audit `AUTH_VERIFY_LOCKED`, return 429.
   c. `AuthService.lookupByPhone(phone_number)` — calls SECURITY DEFINER `auth_lookup_user_by_phone(phone)`. Returns `(shop_id, user_id, role, status, firebase_uid)` or empty. No match → increment `verify_failures`, audit `AUTH_VERIFY_FAILURE`, return 403 `auth.not_provisioned`.
   d. Match but `status != 'ACTIVE' AND status != 'INVITED'` → audit `AUTH_VERIFY_REJECTED`, return 403.
   e. Match found. If `firebase_uid` is null (first-time link, status was `INVITED`): server builds a provisional `TenantContext` from the SECURITY-DEFINER result and calls `tenantContext.runWith({shopId, tenant}, () => withTenantTx(async tx => ...))` — this temporarily seeds ALS for the scope of the link-UPDATE transaction only, same mechanism the TenantInterceptor uses on authenticated requests. Inside: UPDATE `shop_users` SET `firebase_uid = $uid, status = 'ACTIVE', activated_at = now()` WHERE id = $user_id. Audit `AUTH_USER_PROVISIONED`. ALS unwinds on transaction commit; the outer `@SkipTenant` endpoint continues tenant-less. No new `withExplicitTenantTx` API is introduced — single-surface discipline preserved.
   f. If `firebase_uid` exists but doesn't match the incoming Firebase `uid` — this is a phone re-registration or collision; audit `AUTH_UID_MISMATCH`, return 403 `auth.uid_mismatch`. Admin intervention required (runbook entry).
   g. Set Firebase custom claims via Admin SDK: `setCustomUserClaims(uid, {shop_id, role})`. Claims propagate to the next ID token refresh (up to ~1hr lag).
   h. Reset `auth_rate_limits` row for this phone (`verify_failures = 0, locked_until = null`).
   i. Audit `AUTH_VERIFY_SUCCESS`.
   j. Return 200 `{user: {id, display_name, role}, tenant: {id, slug, display_name, config}, requires_token_refresh: true}`.
8. Client calls `getIdToken(true)` to force-refresh and pick up the new custom claims, stores nothing custom (Firebase SDK handles persistence).
9. App navigates to `(tabs)/index.tsx` — the empty dashboard. Renders brand top bar, Hindi hero copy, two CTAs.
10. All subsequent requests go through `TenantInterceptor`: `FirebaseJwtStrategy` verifies the token, extracts `shop_id` from custom claims, `DrizzleTenantLookup` validates tenant ACTIVE, `TenantContext` is constructed with `{shopId, tenant, userId, role}` populated, `tenantContext.runWith(ctx, handler)` seeds ALS for the request. `withTenantTx` in every service method issues `SET LOCAL app.current_shop_id`, RLS enforces.

### 4.2 Tenant resolution order (E2-S1 deferral closed)

Order before this story: Host → Header → JWT. Problem: a raw `X-Tenant-Id` header would shadow a verified JWT claim.

Order after this story: **JWT → Host → Header**. With additional rule: if a verified JWT carries `shop_id` custom claim AND `X-Tenant-Id` header is present AND they disagree, return 403 `tenant.claim_conflict` + audit `TENANT_CLAIM_CONFLICT`. Raw header is only consulted when no JWT is present (unauthenticated endpoints only — and there are only two: `/healthz`, `/tenant/boot`).

**Request-pipeline ordering is load-bearing.** `FirebaseJwtGuard` (Passport-based) runs *before* `TenantInterceptor` in the NestJS chain. The guard calls `verifyIdToken(token, checkRevoked=true)` exactly once and attaches the decoded payload to `request.user = {uid, phone_number, shop_id, role}`. The interceptor then runs and `HttpTenantResolver.fromJwt(req)` reads `req.user?.shop_id` (no re-verification, no JWKS re-fetch, no duplicate Admin SDK call). If `req.user` is absent (the request bypassed the guard via `@SkipAuth`), `fromJwt` returns `undefined` and the resolver falls through to `fromHost` / `fromHeader`. Wiring in `AppModule`: `APP_GUARD` registers `FirebaseJwtGuard` (globally; skippable via `@SkipAuth`), `APP_INTERCEPTOR` registers `ConditionalTenantInterceptor` (globally; skippable via `@SkipTenant`) — guard-before-interceptor order is guaranteed by NestJS's fixed pipeline.

### 4.3 GlobalExceptionFilter hardening (E2-S1 deferral closed)

Current behavior: HttpException → RFC 7807 problem+json. Non-HttpException (e.g., Postgres unique-constraint violation on `shop_users.phone`, Drizzle serialization error) → NestJS default handler leaks error message into response body.

New behavior: non-HttpException → 500 `internal_error` with `{detail: 'An unexpected error occurred.', requestId: <uuid>}`. Full error object goes to the Pino logger (already PII-scrubbed by E2-S1 `redactPii`). Sentry captures with scope-scrubbed payload. The `requestId` is attached to both response and log line so support can correlate.

Special-case: `pg` constraint violation codes (`23505` unique, `23503` FK, `23514` check) are mapped to structured 409/422 RFC 7807 responses **without** the raw message. For example, unique-phone collision on `shop_users` returns `{type: 'conflict', title: 'phone_already_registered', status: 409}` — no phone number in response body.

### 4.4 RLS fail-loud (E2-S1 deferral closed)

Current migration:
```sql
USING (shop_id = current_setting('app.current_shop_id', true)::uuid)
```
The `true` second arg = "missok": if the GUC is unset, `current_setting` returns empty string, cast fails silently, policy returns false, query returns zero rows. Leaks as "no data" rather than "no tenant context".

New migration (shipped in this story via `0003_rls_fail_loud.sql`):
```sql
USING (shop_id = current_setting('app.current_shop_id')::uuid)
```
Missing GUC → Postgres raises `42704 unrecognized configuration parameter`. `withTenantTx` ALWAYS sets the GUC via `SET LOCAL`; connection-pool on-checkout still sets the poison default (`00000000-0000-0000-0000-000000000000`) so any code path that somehow bypasses `withTenantTx` fails loud at either the pool level (cast works, no rows match) or the query level (if the default somehow got RESET) — not silent-empty.

Invariant #12 in E2-S1's spec (poison-default returns 0 rows) remains true. New invariant added: "Connection without poison default + without SET LOCAL raises 42704" — tested in the tenant-isolation harness.

### 4.5 Endpoint-walker lit up (E2-S1 deferral closed)

E2-S1 shipped the `endpoint-walker` scaffold but it only exercised `/healthz` (which is `@SkipTenant`, useless for cross-tenant assertion). This story is the first with a real tenant-scoped endpoint: `GET /api/v1/auth/me`.

Walker additions:
- Provisions 3 Firebase users with distinct phones (test numbers `+15555550001..0003`), seeds 3 `shops` rows (A, B, C) and 3 `shop_users` rows linked to the Firebase UIDs.
- Asserts: A's Firebase ID token → `/auth/me` returns A's row only; B's token → B's row only; C's token → C's row only. No cross-read.
- Asserts: A's token + `X-Tenant-Id: B` header → 403 `tenant.claim_conflict` (new JWT-wins rule exercised).
- Asserts: no token at all → 401 `auth.missing`.
- Asserts: malformed token → 401 `auth.invalid`.
- Asserts: valid token for a `SUSPENDED` shop → 403 `tenant.inactive`.

These assertions run in CI as part of `turbo run test:tenant-isolation`.

## 6. Data model changes

### 6.1 `0002_auth_link.sql`

```sql
-- Extend shop_users
ALTER TABLE shop_users ADD COLUMN firebase_uid TEXT;
CREATE UNIQUE INDEX shop_users_firebase_uid_uq ON shop_users (firebase_uid) WHERE firebase_uid IS NOT NULL;

-- auth_rate_limits: pre-tenant-resolution, platform-global
CREATE TABLE auth_rate_limits (
  phone_e164 TEXT PRIMARY KEY,
  verify_failures INT NOT NULL DEFAULT 0,
  window_started_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  locked_until TIMESTAMPTZ,
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
-- Not ENABLE ROW LEVEL SECURITY — this is platformGlobalTable, correctly.
-- Mirrors existing convention for `shops`.

-- platform_audit_events: pre-tenant-resolution audit trail for auth flows
-- (referenced by §10; separate from the tenant-scoped audit_events table).
CREATE TABLE platform_audit_events (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  action TEXT NOT NULL,
  ip_address INET,
  user_agent TEXT,
  request_id TEXT,
  phone_hash TEXT,                 -- SHA-256 of phone_e164; NEVER raw phone
  metadata JSONB NOT NULL DEFAULT '{}',
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX platform_audit_events_action_created_at_idx
  ON platform_audit_events (action, created_at DESC);
-- Append-only; same grant pattern as tenant-scoped audit_events:
REVOKE UPDATE, DELETE ON platform_audit_events FROM app_user;
GRANT INSERT, SELECT ON platform_audit_events TO app_user;

-- SECURITY DEFINER: phone → tenant resolution (pre-auth, no tenant ctx)
CREATE OR REPLACE FUNCTION auth_lookup_user_by_phone(p_phone TEXT)
RETURNS TABLE(
  shop_id UUID, user_id UUID, role shop_user_role,
  status shop_user_status, firebase_uid TEXT
)
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = public, pg_temp
AS $$
BEGIN
  RETURN QUERY
    SELECT su.shop_id, su.id, su.role, su.status, su.firebase_uid
    FROM shop_users su
    JOIN shops s ON s.id = su.shop_id
    WHERE su.phone = p_phone
      AND s.status = 'ACTIVE'
    LIMIT 1;  -- unique (shop_id, phone); safe
END
$$;
REVOKE ALL ON FUNCTION auth_lookup_user_by_phone(TEXT) FROM PUBLIC;
GRANT EXECUTE ON FUNCTION auth_lookup_user_by_phone(TEXT) TO app_user;
ALTER FUNCTION auth_lookup_user_by_phone(TEXT) OWNER TO platform_admin;

-- SECURITY DEFINER: slug → tenant bootstrap (pre-auth, brand lookup)
CREATE OR REPLACE FUNCTION tenant_boot_lookup(p_slug TEXT)
RETURNS TABLE(id UUID, display_name TEXT, config JSONB)
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = public, pg_temp
AS $$
BEGIN
  RETURN QUERY
    SELECT s.id, s.display_name, s.config
    FROM shops s
    WHERE s.slug = p_slug AND s.status = 'ACTIVE'
    LIMIT 1;
END
$$;
REVOKE ALL ON FUNCTION tenant_boot_lookup(TEXT) FROM PUBLIC;
GRANT EXECUTE ON FUNCTION tenant_boot_lookup(TEXT) TO app_user;
ALTER FUNCTION tenant_boot_lookup(TEXT) OWNER TO platform_admin;
```

## 9. Rate-limit model (NFR-S4)

| Event | Table mutation | Behavior |
|---|---|---|
| `/auth/session` with non-existent phone | `verify_failures += 1`, `window_started_at = now()` if window expired | Returns 403 `auth.not_provisioned` |
| `/auth/session` with expired/invalid Firebase token | `verify_failures += 1` | Returns 401 `auth.token_invalid` |
| `/auth/session` success | Delete row (or zero out) | 200 |
| `verify_failures` reaches 5 within 15-min window | No state change; server continues | Toast shown to client: "5 असफल प्रयास — सावधान रहें" |
| `verify_failures` reaches 10 within 15-min window | `locked_until = now() + 15 minutes` | All further attempts return 429 `auth.locked` until `locked_until` passes |

Window is rolling — on each failure, if `now() - window_started_at > 15 minutes`, reset counter to 1 and restart the window.

Semgrep + Drizzle enforce that all writes to `auth_rate_limits` go through `AuthRateLimitService` (compiled rule: `goldsmith/no-raw-rate-limit-access`).

## 10. Audit events (this story)

All routed via `@goldsmith/audit` `auditLog()`. New enum values:

```ts
enum AuditAction {
  // E2-S1 existing values…

  AUTH_VERIFY_SUCCESS        = 'AUTH_VERIFY_SUCCESS',
  AUTH_VERIFY_FAILURE        = 'AUTH_VERIFY_FAILURE',
  AUTH_VERIFY_LOCKED         = 'AUTH_VERIFY_LOCKED',
  AUTH_VERIFY_REJECTED       = 'AUTH_VERIFY_REJECTED',
  AUTH_USER_PROVISIONED      = 'AUTH_USER_PROVISIONED',
  AUTH_UID_MISMATCH          = 'AUTH_UID_MISMATCH',
  AUTH_TOKEN_INVALID         = 'AUTH_TOKEN_INVALID',
  TENANT_CLAIM_CONFLICT      = 'TENANT_CLAIM_CONFLICT',
  TENANT_BOOT                = 'TENANT_BOOT',
}
```

Audit rows split across two tables along the tenant-resolution boundary:

| Table | Scope | Populated by | Contains |
|---|---|---|---|
| `audit_events` (tenantScopedTable, shipped in E2-S1) | Post-tenant-resolution events | `auditLog()` from `packages/audit` | All `AUTH_VERIFY_SUCCESS` (after tenant resolved), `AUTH_USER_PROVISIONED`, `TENANT_CLAIM_CONFLICT`, and all future tenant-scoped events |
| `platform_audit_events` (platformGlobalTable, shipped in this story) | Pre-tenant-resolution events | `platformAuditLog()` from `packages/audit` | `AUTH_VERIFY_FAILURE`, `AUTH_VERIFY_LOCKED`, `AUTH_VERIFY_REJECTED`, `AUTH_TOKEN_INVALID`, `AUTH_UID_MISMATCH`, `TENANT_BOOT` |

Why split instead of making `audit_events.shop_id` nullable: keeping tenant-scoped auditing strictly tenant-scoped preserves the E2-S1 invariant that *every* row in `audit_events` is visible only to its owning tenant, no RLS special-casing, no `is_platform_admin` escape hatch. The `platform_audit_events` table is small, append-only, visible only to `platform_admin` role (no `app_user` SELECT grant beyond the INSERT path needed for writes). Semgrep rule `goldsmith/platform-audit-skip-tenant-only` enforces that `platformAuditLog()` is only called from handlers marked `@SkipTenant`.

Phone numbers are **NEVER** written raw to either table. Both `audit_events.metadata` and `platform_audit_events.phone_hash` use SHA-256(phone_e164). PII-scrub rule enforced via Semgrep `goldsmith/no-phone-in-audit-metadata` (see §13, invariant 20).

Row shape:
- `audit_events` (unchanged from E2-S1): `{id, shop_id, user_id, action, ip_address, user_agent, request_id, metadata}`
- `platform_audit_events` (new): `{id, action, ip_address, user_agent, request_id, phone_hash, metadata}`

### 11.3 Tenant-isolation (endpoint-walker lit up)

- 3 tenants A/B/C seeded + 3 Firebase test users linked.
- Walker iterates registered routes. For each tenant-scoped route:
  - A's token → returns A's data only (assert at least one row returned)
  - B's token → returns B's data only; no A/C bleed in response body (deep-scan)
  - C's token → returns C's data only

- In this story, only `/auth/me` qualifies; as more endpoints ship, the walker picks them up automatically (registered via a decorator `@TenantWalkerRoute()`).

## 18. Open questions flagged for plan-writing session

The plan session must resolve these before code is written:

1. **Firebase emulator in CI — containerized or SDK-emulator?** The Firebase emulator suite is normally a `firebase emulators:start` process, not a Docker image. For Testcontainers integration, we likely need either a community Docker image or a Node.js subprocess started inside the test setup. Plan session picks one (recommend: subprocess via `firebase-tools` dev dep + Testcontainers-style lifecycle).

2. **Expo SDK 50 vs 51.** SDK 51 released 2024, adds Expo Router v3 and improved `@react-native-firebase` support. Check whether SDK 50 is still the right default or whether we move directly to 51. Plan session confirms.

3. **`@react-native-firebase` requires native modules — Dev Client needed?** Yes — Expo Managed Workflow does not support `@react-native-firebase/auth` directly without Expo Dev Client config plugins. Plan session lock: use Expo Dev Client with config plugin, document Dev Client build step in `docs/runbook.md`.

4. **TenantContext type strictness.** Currently `userId` and `role` are optional (`@sinceStory 1.1`). This story makes them required for authenticated endpoints. Plan session decides: flip to required and thread the optionality via a separate `UnauthenticatedContext` type? Or keep optional and lint-enforce presence at boundaries? Recommend: flip required; add `UnauthenticatedContext` for `@SkipAuth` endpoints.

5. **Tenant-boot cache TTL.** Should `GET /tenant/boot` responses be cached client-side? Shop name/logo won't change often. Plan session: 24h cache in AsyncStorage with `ETag`-based revalidation.

6. **Firebase custom claim propagation lag.** After `setCustomUserClaims`, the client's next `getIdToken(true)` picks up the new claims — but subsequent API calls between step 7g and step 8 in §4.1 will have an ID token WITHOUT claims. The spec says we return `requires_token_refresh: true` and the client force-refreshes; plan session must lock the client error-handling if a claim-less token hits a tenant-required endpoint. Recommend: interceptor returns 401 `auth.claim_propagation`, client auto-retries once with force-refresh.

7. **`@SkipAuth` decorator implementation.** Does it live in `@goldsmith/tenant-context` (consistent with `@SkipTenant`) or in a new `@goldsmith/auth-guards` package? Plan session picks (recommend: add to existing `tenant-context` package; both decorators are request-pipeline-level concerns).

8. **Maestro in CI vs locally only.** Android emulator on GitHub Actions macOS runners adds 4-5 min per PR. Plan session decides: run Maestro in a nightly job + on `[e2e]` PR label, not every PR. Happy path unit/integration coverage plus a static a11y lint makes this acceptable.

9. **Dev vs prod Firebase project split.** Do we provision a real Firebase project now (free Spark tier) to avoid emulator-only development friction? Or stay emulator-only until SOW signs? Recommend: provision `goldsmith-dev` Firebase project now (no Azure subscription needed; Firebase is its own billing surface). Zero recurring cost on Spark tier. Plan session confirms.

10. **Hindi copy ownership.** Who writes the final Hindi strings? `packages/i18n/locales/hi-IN/*.json` for this story covers only ~20 keys. For Story 1.2+ and Epic 3+ the key count explodes. Plan session: this story uses Claude-drafted Hindi; anchor jeweller reviews at alpha; full translation review happens pre-beta (a new BMAD task, not this story).

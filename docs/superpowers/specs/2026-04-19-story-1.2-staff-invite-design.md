# Story 1.2 — Shop Owner invites a staff member by phone and assigns a role

## 1. Summary

Rajesh-ji opens the Staff screen, taps "+ Staff जोड़ें", enters Amit's name, phone, and role, and taps "आमंत्रण भेजें". The server creates an INVITED `shop_users` row, returns a pre-composed Hindi invitation message, and the app opens the native share sheet so Rajesh-ji can forward it via his own WhatsApp in one tap. When Amit opens the shopkeeper app and completes Firebase OTP, the existing `/auth/session` logic (Story 1.1) detects the INVITED row and promotes him to ACTIVE automatically.

## 2. Out of scope

- AiSensy / MSG91 direct-send invite — deferred until anchor MRR justifies Rs 1,500/mo
- `role: shop_admin` invite — restricted at API layer; unlocked in Story 1.3 with full `@Roles()` guard
- Re-invite of REVOKED staff — Story 1.4 (access management)
- Staff profile edit (display_name update post-activation) — Story 1.4+

## 3. Architecture

### 3.1 New module: `apps/api/src/modules/staff/`

```
apps/api/src/modules/staff/
├── staff.module.ts
├── staff.controller.ts     POST /api/v1/staff  (invite)
│                           GET  /api/v1/staff  (list)
├── staff.service.ts        invite(), list()
├── staff.repository.ts     insertInvited(), upsertInvited(), findByPhone(), findAllByShop()
└── dto/
    ├── invite-staff.dto.ts
    └── staff-list-item.dto.ts
```

The `auth` module is unchanged. Staff management is a distinct bounded context; invite, revoke (1.4), list, and future role-change stories all land here.

### 3.2 Request pipeline

```
FirebaseJwtGuard → TenantInterceptor → StaffController → StaffService → StaffRepository
                                                       ↓
                                              withTenantTx → SET LOCAL app.current_shop_id
                                                       ↓
                                              INSERT/UPDATE shop_users (RLS enforced)
                                                       ↓
                                              auditLog(STAFF_INVITED)
                                                       ↓
                                              return { staff, share: { text } }
```

`POST /api/v1/staff` is NOT `@SkipTenant`. It requires a fully authenticated `TenantContext`. `shop_id` for the new row comes exclusively from `TenantContext.shopId` — never from the request body.

### 3.3 Share text generation

Server generates the Hindi invitation string using tenant + staff data:

```ts
function buildShareText(tenantDisplayName: string, staffDisplayName: string, role: ShopUserRole): string {
  const roleLabel = role === 'shop_manager' ? 'Manager' : 'Staff';
  return `${tenantDisplayName} में ${staffDisplayName} को ${roleLabel} के रूप में आमंत्रित किया गया है। Shopkeeper app download करें।`;
}
```

No deep link infrastructure in MVP. `[link]` is omitted until EAS build + per-tenant Play Store entries exist (post-SOW story). The message is human-readable and sufficient for Rajesh-ji to forward verbally or via WhatsApp.

## 4. Data model changes

### 4.1 Schema extension — `packages/db/src/schema/shop-users.ts`

Add two columns:

```ts
invited_by_user_id: uuid('invited_by_user_id'),          // FK to shop_users.id; null for owner row
invited_at: timestamp('invited_at', { withTimezone: true }),
```

Existing columns (`status`, `firebase_uid`, `activated_at`) are unchanged from Story 1.1.

### 4.2 Migration — `packages/db/src/migrations/0004_staff_invite.sql`

```sql
ALTER TABLE shop_users
  ADD COLUMN invited_by_user_id UUID REFERENCES shop_users(id) ON DELETE SET NULL,
  ADD COLUMN invited_at TIMESTAMPTZ;

-- Back-fill: existing owner rows get invited_at = created_at
UPDATE shop_users SET invited_at = created_at WHERE invited_at IS NULL;
```

No new unique index needed — `(shop_id, phone)` uniqueness is already enforced from E2-S1.

## 5. API contract

### 5.1 `POST /api/v1/staff` — invite staff member

**Auth:** Firebase ID token required. `TenantContext.role` must be `shop_admin` (inline check in `StaffService.invite()` until Story 1.3's `@Roles()` guard).

**Request body (`InviteStaffDto`):**
```ts
{
  phone: string;        // E.164, must start with +91, 10 digits after prefix
  display_name: string; // non-empty, max 100 chars
  role: 'shop_manager' | 'shop_staff';  // shop_admin excluded via IsEnum
}
```

**Happy-path response `201`:**
```json
{
  "staff": {
    "id": "<uuid>",
    "phone": "+91XXXXXXXXXX",
    "display_name": "Amit",
    "role": "shop_staff",
    "status": "INVITED",
    "invited_at": "<iso8601>"
  },
  "share": {
    "text": "राजेश जी की ज्वैलर्स में Amit को Staff के रूप में आमंत्रित किया गया है। Shopkeeper app download करें।"
  }
}
```

**Re-invite (INVITED → INVITED):** Returns `201` with updated `invited_at` and fresh `share.text`. Repository does `UPDATE ... SET invited_by_user_id, invited_at WHERE shop_id = $shopId AND phone = $phone AND status = 'INVITED'`.

**Error responses:**

| Condition | Status | Code |
|---|---|---|
| `role: shop_admin` in body | 400 | `staff.role_not_allowed` |
| Phone already ACTIVE | 409 | `staff.already_exists` |
| Phone already SUSPENDED | 409 | `staff.already_exists` |
| Caller is not `shop_admin` | 403 | `auth.forbidden` |
| Invalid phone format | 400 | `validation.phone_invalid` |

### 5.2 `GET /api/v1/staff` — list staff

**Auth:** Firebase ID token required. Any role can list (owner + manager + staff all see the list).

**Response `200`:**
```json
{
  "staff": [
    {
      "id": "<uuid>",
      "display_name": "Amit",
      "phone_last4": "3210",
      "role": "shop_staff",
      "status": "INVITED",
      "invited_at": "<iso8601>",
      "activated_at": null
    }
  ]
}
```

`phone_last4` only — never full phone in list response (PII minimisation).

## 7. Audit events

New action in `packages/audit`:

```ts
STAFF_INVITED = 'STAFF_INVITED',
```

Routed to tenant-scoped `audit_events` table (actor has full `TenantContext` at invite time).

```json
{
  "actor_user_id": "<inviting owner's user_id>",
  "action": "STAFF_INVITED",
  "subject_type": "shop_user",
  "subject_id": "<new shop_user id>",
  "before": null,
  "after": { "role": "shop_staff", "status": "INVITED", "display_name": "Amit" },
  "metadata": { "request_id": "<uuid>" }
}
```

Re-invite fires a second `STAFF_INVITED` with `before: { status: "INVITED", invited_at: "<old>" }`.

**PII rule:** `phone` is NEVER written to audit `after` or `metadata`. Enforced by existing Semgrep rule `goldsmith/no-phone-in-audit-metadata`.

## 9. Tests required

### 9.1 Unit

- `StaffService.invite()` — happy path creates INVITED row + returns share text; duplicate ACTIVE phone → 409; duplicate INVITED phone → 201 re-invite; `shop_admin` role in body → 400; non-`shop_admin` caller → 403.
- `StaffService.list()` — returns masked phone_last4 only; no cross-tenant rows.
- `buildShareText()` — correct Hindi string for each role variant.

### 9.2 Integration (Testcontainers Postgres + full NestJS app)

- Invite flow end-to-end: POST `/staff` → assert `shop_users.status = 'INVITED'`, `invited_by_user_id` populated, `invited_at` set, `STAFF_INVITED` audit row inserted, `share.text` contains tenant display name.
- Re-invite: POST twice with same phone → second call returns 201, `invited_at` updated, one `shop_users` row only.
- ACTIVE conflict: seed ACTIVE row → POST → 409 `staff.already_exists`.
- Role restriction: POST with `role: shop_admin` → 400 `staff.role_not_allowed`.
- Non-owner caller: POST with `shop_manager` token → 403 `auth.forbidden`.
- List: GET `/staff` → `phone_last4` only, full phone absent from response body.

## 11. Acceptance criteria (BDD)

**Staff screen navigation:**
- **Given** the owner is on the empty dashboard (Story 1.1), **when** they tap "Staff जोड़ें", **then** the Staff screen opens showing `आपके Staff (0)` and the empty-state copy in Hindi.

**Happy-path invite:**
- **Given** the owner fills name `Amit`, phone `+919876543211`, role `Staff`, **when** they tap "आमंत्रण भेजें", **then** the server returns 201, `shop_users` has a row with `status=INVITED`, `invited_by_user_id` is the owner's user_id, `STAFF_INVITED` audit event is written, the native share sheet opens with a Hindi message containing the tenant name and staff name, and the list updates optimistically to show `Amit — स्टाफ — आमंत्रित`.

**Role restriction:**
- **Given** the invite DTO contains `role: shop_admin`, **when** the request reaches the server, **then** 400 `staff.role_not_allowed` is returned before any DB write.

**Duplicate ACTIVE:**
- **Given** a phone already exists as ACTIVE staff, **when** the owner submits the invite form with that phone, **then** the server returns 409 and the UI shows `यह नंबर पहले से staff में शामिल है।` inline.

**Re-invite:**
- **Given** a phone exists with `status=INVITED`, **when** the owner submits again with the same phone, **then** 201 is returned, `invited_at` is updated, one `shop_users` row exists, and a new `STAFF_INVITED` audit event is written.

**Tenant isolation:**
- **Given** CI runs the tenant-isolation suite, **when** tenant A's owner invites staff, **then** that staff row is not visible in tenant B's `GET /api/v1/staff` response and all prior walker assertions still pass.

**PII:**
- **Given** a `STAFF_INVITED` audit event, **when** the row is inspected, **then** no `phone` field appears in `after` or `metadata`.

**Accessibility:**
- **Given** axe-core-rn runs on the staff screen and invite sheet, **then** zero WCAG 2.1 AA violations.
- **Given** ESLint runs `goldsmith/no-untranslated-strings`, **then** zero bare string literals in staff screen JSX.

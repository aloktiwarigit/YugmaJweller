## 2. Architectural Decisions

### 2.1 PolicyGuard vs RolesGuard — Layer, Don't Replace

RolesGuard provides coarse role-level gating via `@Roles('shop_admin')`. PolicyGuard provides fine-grained permission-key checks via `@Permission('billing.void')`. Both are registered as global guards in `AppModule` (RolesGuard first, PolicyGuard second). Both can decorate the same endpoint when needed. shop_admin always short-circuits PolicyGuard — no DB/cache lookup ever runs for admin.

This preserves backward compatibility with all existing `@Roles()` decorations and lets Story 1.4 introduce permission-gated endpoints cleanly without touching the existing guard.

### 2.2 Permission Cache Invalidation — Per-Role

Cache key: `shop:{shopId}:permissions:{role}` (e.g. `shop:abc123:permissions:shop_manager`).  
On PUT /auth/roles/:role/permissions, invalidate only that role's key. shop_admin has no cache entry (bypassed). The 60s TTL bounds worst-case staleness to 60s for any missed invalidation.

### 2.3 SettingsGroupCard — Non-expandable Card Group

Props: `{ title: string; children: React.ReactNode; testID?: string }`. No accordion state. Accordion is a separate concern; the spec does not require expand/collapse in Story 1.4. The component renders a titled bordered card section.

```sql
CREATE TABLE role_permissions (
  id             UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
  shop_id        UUID        NOT NULL REFERENCES shops(id),
  role           shop_user_role NOT NULL,
  permission_key TEXT        NOT NULL,
  is_enabled     BOOLEAN     NOT NULL DEFAULT false,
  updated_at     TIMESTAMPTZ NOT NULL DEFAULT now(),
  UNIQUE(shop_id, role, permission_key)
);

ALTER TABLE role_permissions ENABLE ROW LEVEL SECURITY;
ALTER TABLE role_permissions FORCE ROW LEVEL SECURITY;

CREATE POLICY rls_role_permissions_tenant_isolation ON role_permissions
  FOR ALL
  USING      (shop_id = current_setting('app.current_shop_id')::uuid)
  WITH CHECK (shop_id = current_setting('app.current_shop_id')::uuid);

GRANT SELECT, INSERT, UPDATE ON role_permissions TO app_user;
```

## 5. Part B — Role Permissions API

### 5.1 PermissionsRepository

New class in `apps/api/src/modules/auth/permissions.repository.ts`:

```typescript
getPermissions(shopId: string, role: ShopUserRole): Promise<Record<string, boolean>>
upsertPermission(shopId: string, role: ShopUserRole, key: string, enabled: boolean): Promise<void>
seedDefaults(shopId: string): Promise<void>  // called at shop creation
```

`getPermissions` runs under tenant context, returns all rows for the shop+role as a `Record<permissionKey, isEnabled>`.  
`upsertPermission` uses `INSERT ... ON CONFLICT DO UPDATE SET is_enabled = $4, updated_at = now()`.

### 5.2 AuthController additions

```typescript
@Get('/roles/:role/permissions')
@Roles('shop_admin')
async getRolePermissions(@Param('role') role: string, @TenantContextDec() ctx: AuthenticatedTenantContext)

@Put('/roles/:role/permissions')
@Roles('shop_admin')
async updateRolePermission(@Param('role') role: string, @Body() dto: UpdatePermissionDto, @TenantContextDec() ctx: AuthenticatedTenantContext)
```

`UpdatePermissionDto`:
```typescript
class UpdatePermissionDto {
  @IsString() permission_key: string;
  @IsBoolean() is_enabled: boolean;
}
```

GET response: `{ role, permissions: Record<string, boolean> }`  
PUT flow:
1. If `role === 'shop_admin'` → 403 `{ code: 'auth.admin_permissions_immutable' }`.
2. `permissionsRepo.upsertPermission(...)`.
3. `permissionsCache.invalidate(shopId, role)`.
4. `auditLog(PERMISSIONS_UPDATED, { before: { role, key, was: oldValue }, after: { role, key, is: newEnabled } })`.
5. Return `204 No Content`.

## 6. Part C — PolicyGuard + PermissionsCache

### 6.1 PermissionsCache

`packages/tenant-config/src/permissions-cache.ts`:

```typescript
export class PermissionsCache {
  private static readonly TTL_SEC = 60;
  constructor(private readonly redis: Redis) {}

  async getPermissions(shopId: string, role: ShopUserRole): Promise<Record<string, boolean> | null>
  async setPermissions(shopId: string, role: ShopUserRole, data: Record<string, boolean>): Promise<void>
  async invalidate(shopId: string, role: ShopUserRole): Promise<void>
  private key(shopId: string, role: ShopUserRole): string  // `shop:${shopId}:permissions:${role}`
}
```

`getPermissions`: parse JSON, return null on error/miss (same swallow-pattern as SettingsCache).  
`setPermissions`: `SET key JSON EX 60`.  
`invalidate`: `DEL key`.

### 6.2 @Permission() Decorator

`apps/api/src/common/decorators/permission.decorator.ts`:
```typescript
export const PERMISSION_KEY = 'permission';
export const Permission = (key: string): MethodDecorator => SetMetadata(PERMISSION_KEY, key);
```

### 6.3 PolicyGuard

`apps/api/src/modules/auth/guards/policy.guard.ts` (auth module, not common — avoids circular dependency since it injects PermissionsRepository which lives in AuthModule). PolicyGuard is provided by AuthModule and registered globally via `APP_GUARD` token in AppModule.

```typescript
@Injectable()
export class PolicyGuard implements CanActivate {
  constructor(
    private readonly reflector: Reflector,
    private readonly permissionsCache: PermissionsCache,
    private readonly permissionsRepo: PermissionsRepository,
  ) {}

  async canActivate(ctx: ExecutionContext): Promise<boolean> {
    const requiredKey = this.reflector.get<string | undefined>(PERMISSION_KEY, ctx.getHandler());
    if (!requiredKey) return true;                       // no @Permission — pass

    const tc = tenantContext.current();
    if (!tc?.authenticated) throw new UnauthorizedException({ code: 'auth.not_authenticated' });
    if (tc.role === 'shop_admin') return true;           // admin bypass

    let perms = await this.permissionsCache.getPermissions(tc.shopId, tc.role);
    if (!perms) {
      perms = await this.permissionsRepo.getPermissions(tc.shopId, tc.role);
      await this.permissionsCache.setPermissions(tc.shopId, tc.role, perms);
    }

    const allowed = perms[requiredKey] ?? false;         // deny-by-default
    if (!allowed) throw new ForbiddenException({
      errorCode: 'auth.permission_denied',
      permissionKey: requiredKey,
    });
    return true;
  }
}
```

PolicyGuard is registered **after** RolesGuard in `AppModule` providers array so role-level checks fire first.

## 2. Data Layer

### 2.1 Migration `0006_shop_settings.sql`

```sql
-- 0006_shop_settings.sql
-- Extends shops with mutable profile columns; creates shop_settings (Epic 2 config home).

-- Part A: profile columns on shops (globally-queryable identity fields)
ALTER TABLE shops
  ADD COLUMN address_json          JSONB,
  ADD COLUMN gstin                 TEXT,
  ADD COLUMN bis_registration      TEXT,
  ADD COLUMN contact_phone         TEXT,
  ADD COLUMN operating_hours_json  JSONB,
  ADD COLUMN about_text            TEXT,
  ADD COLUMN logo_url              TEXT,
  ADD COLUMN years_in_business     INT;
```

All columns are nullable — shops may not fill every field on day 1.  
`display_name` (already on `shops`) is the shop name — the API maps it to/from `name` in the DTO.  
`contact_phone` is the shop's published directory number, separate from owner's login phone in `shop_users`.

```sql
-- Part B: shop_settings — one row per shop; home for Epic 2 config/preference columns
CREATE TABLE shop_settings (
  shop_id                   UUID PRIMARY KEY REFERENCES shops(id) ON DELETE RESTRICT,
  making_charges_json       JSONB,
  wastage_json              JSONB,
  loyalty_json              JSONB,
  rate_lock_days            INT,
  try_at_home_enabled       BOOLEAN NOT NULL DEFAULT FALSE,
  try_at_home_max_pieces    INT,
  custom_order_policy_text  TEXT,
  return_policy_text        TEXT,
  notification_prefs_json   JSONB,
  updated_at                TIMESTAMPTZ NOT NULL DEFAULT now()
);

ALTER TABLE shop_settings ENABLE ROW LEVEL SECURITY;
ALTER TABLE shop_settings FORCE ROW LEVEL SECURITY;
CREATE POLICY rls_shop_settings_tenant_isolation ON shop_settings
  FOR ALL
  USING      (shop_id = current_setting('app.current_shop_id')::uuid)
  WITH CHECK (shop_id = current_setting('app.current_shop_id')::uuid);

GRANT SELECT, INSERT, UPDATE ON shop_settings TO app_user;
```

**Why `shop_id` as PK directly (not via `tenantScopedTable` helper):**  
`tenantScopedTable` adds `shop_id` as a non-null column + index. Here `shop_id` IS the PK (one row per shop, upsert semantics). Using the helper would create a duplicate `shop_id` column. The Drizzle schema is defined manually with explicit PK FK.

**Why no DELETE grant on `shop_settings`:**  
Settings rows are always upserted, never deleted. Removing the grant prevents accidental hard-deletes; rows survive if a shop is suspended.

  contact_phone: string | null;
  operating_hours_json: Record<string, OperatingHoursDay> | null;

    private readonly ttlSec = 60,

## 4. New Package: `packages/shared`

Zod validation schemas shared across API + future mobile client-side validation.

### `packages/shared/src/schemas/shop-profile.schema.ts`

```typescript
const GSTIN_REGEX = /^[0-9]{2}[A-Z]{5}[0-9]{4}[A-Z]{1}[1-9A-Z]{1}Z[0-9A-Z]{1}$/;

const AddressSchema = z.object({
  street:   z.string().max(200),
  city:     z.string().max(100),
  state:    z.string().max(100),
  pin_code: z.string().regex(/^\d{6}$/, 'PIN must be 6 digits'),
});

const OperatingHoursDaySchema = z.object({
  enabled: z.boolean(),
  open:    z.string().regex(/^\d{2}:\d{2}$/).nullable(),
  close:   z.string().regex(/^\d{2}:\d{2}$/).nullable(),
});

const OperatingHoursSchema = z.object({
  mon: OperatingHoursDaySchema, tue: OperatingHoursDaySchema,
  wed: OperatingHoursDaySchema, thu: OperatingHoursDaySchema,
  fri: OperatingHoursDaySchema, sat: OperatingHoursDaySchema,
  sun: OperatingHoursDaySchema,
});

export const PatchShopProfileSchema = z.object({
  name:               z.string().min(1).max(120).optional(),
  address:            AddressSchema.optional(),
  gstin:              z.string().regex(GSTIN_REGEX, 'GSTIN_INVALID').optional().nullable(),
  bis_registration:   z.string().max(50).optional().nullable(),
  contact_phone:      z.string().regex(/^\+91[6-9]\d{9}$/).optional().nullable(),
  operating_hours:    OperatingHoursSchema.optional(),
  about_text:         z.string().max(500).optional().nullable(),
  logo_url:           z.string().url().optional().nullable(),
  years_in_business:  z.number().int().min(0).max(200).optional().nullable(),
}).strict();

export type PatchShopProfileDto = z.infer<typeof PatchShopProfileSchema>;
```

**Why GSTIN_INVALID code (not inline message):** The error code is translated by the i18n layer per locale. The API returns `{ field: 'gstin', code: 'GSTIN_INVALID' }` — the mobile client maps the code to the Hindi message.

Access matrix for settings endpoints:

| Role | GET /profile | PATCH /profile | POST /logo-upload-url |
|---|---|---|---|
| `shop_admin` | ✅ | ✅ | ✅ |
| `shop_manager` | ✅ | ❌ 403 | ❌ 403 |
| `shop_staff` | ❌ 403 | ❌ 403 | ❌ 403 |

**Domain rationale:** In Indian jewellery retail, only the owner (shop_admin) changes shop-identity information. A manager might need to look up the shop's GSTIN or operating hours for a customer query, hence read access. Staff have no settings access.

### 5.3 Endpoints

#### `GET /api/v1/settings/profile`

Returns the current shop profile from Redis cache (60s TTL); falls back to DB on cache miss.  
Response includes `ETag` header: `"${sha256(responseBody).slice(0, 16)}"`.

```typescript
// Response body
{
  name: string;
  address: AddressDto | null;
  gstin: string | null;
  bis_registration: string | null;
  contact_phone: string | null;
  operating_hours: OperatingHoursDto | null;
  about_text: string | null;
  logo_url: string | null;
  years_in_business: number | null;
  updated_at: string; // ISO-8601
}
```

#### `PATCH /api/v1/settings/profile`

Partial update — any subset of profile fields.  
Save-on-blur: every field focus-out triggers an independent PATCH.

Flow:
1. Validate body with `PatchShopProfileSchema` → 400 with `{ field, code }` array on error
2. Load current profile from DB (for audit `before`)
3. Build SQL UPDATE on `shops` (only columns present in payload)
4. Invalidate `SettingsCache` (Redis `shop:{shopId}:settings:profile`)
5. Invalidate `DrizzleTenantLookup` (in-process tenant cache) — needed because `display_name` is part of the tenant record
6. Emit `auditLog('SETTINGS_PROFILE_UPDATED', { before, after })`
7. Return 200 with full updated profile + new `ETag`

**Why dual-invalidation (step 4 + 5):**  
`DrizzleTenantLookup` is an in-process Map cache used by `TenantInterceptor` to resolve the tenant on every request. It caches `display_name`. When the shopkeeper changes the shop name, the next tenant resolution would serve the stale name for up to 60s unless we invalidate it. `DrizzleTenantLookup.invalidate(shopId)` is already implemented — we just need to inject it into `SettingsService`.

#### `POST /api/v1/settings/profile/logo-upload-url`

Returns a pre-signed Azure Blob Storage SAS URL.  
Client uploads directly to Azure Blob; after upload, client calls `PATCH /settings/profile` with `{ logo_url: "<cdn-url>" }`.

```typescript
// Response
{
  upload_url: string;        // SAS URL (15-min TTL)
  cdn_url: string;           // ImageKit CDN URL the client should send back in the PATCH
  blob_path: string;         // tenants/{shopId}/logo/{uuid}.webp
  expires_at: string;        // ISO-8601
}
```

**`BlobStorageService`:**
- Uses `@azure/storage-blob` `BlobServiceClient` + `generateBlobSASQueryParameters`
- `AZURE_STORAGE_CONNECTION_STRING` env var (dev: `UseDevelopmentStorage=true` via Azurite)
- EXIF stripping: handled server-side via `sharp` after client upload confirmation (follow-up; noted as TODO in the controller)
- Tests mock `BlobServiceClient` via `vi.fn()` / provider override

## 6. Cache Strategy

| Cache | Key | TTL | Invalidated by |
|---|---|---|---|
| `SettingsCache` (Redis) | `shop:{shopId}:settings:profile` | 60s | `PATCH /settings/profile` |
| `DrizzleTenantLookup` (in-process Map) | `shopId` | 60s | `PATCH /settings/profile` (when `name` changes) |

**TTL rationale:** NFR-P6 requires 30-second propagation SLA. A 60s TTL means worst-case 60s until a cache miss triggers a DB re-read. The integration test verifies the guaranteed path: invalidate-on-write means re-fetch after PATCH always sees the new value immediately (TTL is not the propagation mechanism; explicit invalidation is).

**Future:** When the API scales to multiple instances, `DrizzleTenantLookup` in-process cache will diverge across pods. Migration path (not this story): replace with Redis-backed `TenantCache` from `packages/tenant-context`. The interface is already `TenantLookup`, so the swap is a provider change in `AppModule`.

## 9. Testing Plan

### 9.1 Unit — `SettingsService`

| Test | Assertion |
|---|---|
| Valid PATCH name | `repo.updateShopProfile` called with mapped payload; cache invalidated; auditLog called with before/after |
| Invalid GSTIN | Zod throws before reaching repo; returns `{ field: 'gstin', code: 'GSTIN_INVALID' }` |

Run against real Postgres (same pattern as existing audit integration tests).

| Test | Flow |
|---|---|

## 12. Key Invariants (must hold after implementation)

1. Every PATCH that changes `display_name` invalidates BOTH `SettingsCache` and `DrizzleTenantLookup`.
2. No PATCH can write to a different tenant's `shops` row — RLS is the last line of defense; tenant context must be set before every `withTenantTx` call.
3. `shop_settings` row is created via UPSERT on first PATCH — never assumed to pre-exist.
4. Audit `before` snapshot is read inside the same `withTenantTx` transaction as the UPDATE — no TOCTOU window.
5. `BlobStorageService.generateSasUrl()` is injectable/mockable — no static calls to Azure SDK.
6. The `RolesGuard` is applied at the controller method level (not class level) to allow per-endpoint access matrices for future stories in the same controller.

# Story 1.5 — Staff Revocation Design

**Date:** 2026-04-20  
**Class:** A (auth endpoint + Firebase token revocation)  
**Branch:** `feat/story-1.5-staff-revocation`

---

## What we're building

A shop owner can revoke a staff member's access. Revocation is immediate and permanent: the staff member's app session is terminated via Firebase token invalidation, and they cannot log in again.

---

## Design decisions

### 1. `revokeStaff()` lives in `auth.service.ts`

`invite()` already lives in `auth.service.ts`. Putting `revokeStaff()` there keeps the full staff lifecycle (invite → revoke) co-located. Splitting to a `staff.service.ts` without moving `invite()` would create inconsistent responsibility across two services — worse than a slightly larger file (~253 lines total).

### 2. Cross-tenant miss returns 404, not 403

The repository query always includes `AND shop_id = $callerShopId`. Any miss (userId not found, userId belongs to another tenant) returns 0 rows — caller gets 404. This prevents cross-tenant user-existence disclosure: a 403 would confirm the UUID exists somewhere.

### 3. Fetch-then-update (two queries), not UPDATE-then-check

Self-revoke (400) and admin-revoke (403) checks happen **before** the UPDATE. This avoids partially-written revocations and produces clean, independently-testable guard logic. Two queries are cheap; the alternative couples guard semantics to UPDATE result columns.

## API

### Endpoint

```
DELETE /api/v1/auth/staff/:userId
Guards: FirebaseJwtGuard → TenantInterceptor → @Roles('shop_admin')
Response: 204 No Content
```

### `AuthRepository.revokeStaff()`

```typescript
revokeStaff(shopId: string, targetUserId: string): Promise<{ firebaseUid: string | null; role: ShopUserRole } | null>
```

Single `SELECT` scoped to `shop_id`:

```sql
SELECT id, firebase_uid, role
  FROM shop_users
 WHERE id = $1 AND shop_id = $2
```

Returns `null` if 0 rows (tenant-isolation + not-found merged into one path).

Separate `UPDATE` called only after guards pass:

```sql
UPDATE shop_users
   SET status              = 'REVOKED',
       revoked_at          = NOW(),
       revoked_by_user_id  = $revokedBy,
       updated_at          = NOW()
 WHERE id = $1 AND shop_id = $2
```

### `AuthService.revokeStaff()`

```
1. row = repo.revokeStaff(shopId, targetUserId)   -- SELECT only
2. if row === null                                  → NotFoundException 404
3. if targetUserId === callerUserId                 → BadRequestException 400
4. if row.role === 'shop_admin'                     → ForbiddenException 403
5. repo.markRevoked(shopId, targetUserId, callerUserId)  -- UPDATE
6. if row.firebaseUid !== null
     firebase.admin().auth().revokeRefreshTokens(row.firebaseUid)
7. auditLog(STAFF_REVOKED, { before: {status:'ACTIVE'}, after: {status:'REVOKED'} })
8. return void
```

If `firebaseUid` is null (user was invited but never activated), skip step 6. Setting `status='REVOKED'` is sufficient — they can never activate now.

Firebase `revokeRefreshTokens()` invalidates all existing refresh tokens. The JWT strategy already calls `verifyIdToken(token, true)` (checkRevoked: true), so subsequent requests with existing tokens are rejected with `auth/id-token-revoked` → 401.

### `AuthController`

```typescript
@Delete('/staff/:userId')
@Roles('shop_admin')
@HttpCode(204)
async revokeStaff(@Param('userId') userId: string): Promise<void>
```

**On confirm:**
1. `DELETE /api/v1/auth/staff/:userId`
2. Optimistic remove: `setStaff(prev => prev.filter(m => m.id !== target.id))`
3. Toast: "हटा दिया गया" + medium haptic (`Haptics.impactAsync(ImpactFeedbackStyle.Medium)`)
4. On error: revert optimistic update, show error toast:
   - 400: "आप स्वयं को नहीं हटा सकते"
   - 403: "एडमिन को नहीं हटाया जा सकता"

## Smoke test protocol

1. Seed an `ACTIVE` staff user with a known `firebase_uid`
2. `DELETE /api/v1/auth/staff/:userId` with OWNER JWT → expect 204
3. `GET /api/v1/auth/users` → user shows `REVOKED` or is absent
4. `DELETE` same userId again → expect 404
5. `DELETE` with STAFF JWT → expect 403
6. Mobile: "हटाएं" button visible on staff rows; absent on admin row; confirmation sheet appears; toast fires on success

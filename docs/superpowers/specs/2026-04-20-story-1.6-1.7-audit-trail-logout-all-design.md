# Design Spec: Stories 1.6 + 1.7 — Audit Trail Immutability, Read Path & Logout All Devices

**Date:** 2026-04-20  
**Class:** A (auth + security-sensitive)  
**Stories:** 1.6 (audit immutability + read path), 1.7 (logout all devices)  
**Ceremony:** Full — brainstorm → plan → parallel agents → TDD → Codex + /security-review → smoke test

---

## 1. Scope

Both stories share the auth module and land in one PR under a single Codex + `/security-review` gate.

**Story 1.6** establishes three layers of audit immutability (REVOKE, DB trigger, Semgrep), backfills 12 missing `AuditAction` enum values, and adds a paginated audit-log read API + mobile screen.

**Story 1.7** adds `POST /api/v1/auth/logout/all` (Firebase token revocation) and an Account settings screen where the shopkeeper can eject all active sessions.

## 4. Migration — Audit Immutability

**File:** `packages/db/src/migrations/0009_audit_immutability.sql`

> **Renaming note:** If Story 1.5 migrations (0009–0011) land on main before this PR is merged, rename this file to `0012_audit_immutability.sql`. Check `packages/db/src/migrations/` in the target branch before applying.

```sql
-- Layer 1: revoke mutation grants from the app role
REVOKE UPDATE, DELETE ON audit_events FROM app_user;

-- Layer 2: trigger fires for ALL roles, including postgres superuser
CREATE OR REPLACE FUNCTION prevent_audit_mutation()
  RETURNS TRIGGER LANGUAGE plpgsql SECURITY DEFINER AS $$
  BEGIN
    RAISE EXCEPTION 'audit_events is append-only — write via auditLog() only'
      USING ERRCODE = 'restrict_violation';
  END;
$$;

CREATE TRIGGER audit_events_immutable
  BEFORE UPDATE OR DELETE ON audit_events
  FOR EACH ROW EXECUTE FUNCTION prevent_audit_mutation();
```

`SECURITY DEFINER` ensures the trigger body runs under the definer's privileges and fires regardless of the caller's role, closing the superuser bypass gap.

## 5. Semgrep Rule — Audit Immutability

**File:** `ops/semgrep/audit-immutability.yaml`

```yaml
rules:
  - id: no-drizzle-mutation-on-audit-events
    languages: [typescript]
    severity: ERROR
    message: "audit_events is append-only — use auditLog() for writes"
    pattern-either:
      - pattern: $DB.update(auditEvents)
      - pattern: $DB.delete(auditEvents)
```

Catches mutation attempts at CI / pre-push before they ever reach the database.

## 6. Read API

### Endpoint

```
GET /api/v1/auth/audit-log
```

**Auth:** `@Roles('shop_admin', 'shop_manager')` via existing `PolicyGuard`. `shop_staff` → 403 (no special handling needed — RolesGuard rejects before handler).

**Query params:**

| Param | Type | Default | Constraint |
|-------|------|---------|------------|
| `page` | number | 1 | ≥1 |
| `pageSize` | number | 20 | 1–50 (clamped — never 400) |
| `dateRange` | `today\|7d\|30d` | `30d` | unknown value → `30d` |
| `category` | `login\|staff\|permissions\|settings` | (none = all) | unknown value → all |

### Response

```ts
interface AuditLogResponse {
  events: AuditEventDto[];
  total: number;
  page: number;
  pageSize: number;
}

interface AuditEventDto {
  id: string;
  action: string;                          // raw AuditAction enum key; i18n on mobile
  actorName: string;                       // display_name from shop_users JOIN; null → 'System'
  actorRole: string;                       // role from shop_users JOIN; null → 'system'
  createdAt: string;                       // ISO8601 UTC
  metadata?: Record<string, unknown>;
}
```

`action` is the raw enum key. The mobile client owns the Hindi label via `ACTION_HINDI` inline map — same pattern as `ROLE_HINDI` in `staff.tsx`. This keeps the API language-agnostic for the future web admin console (English).

### Category → Actions mapping (in AuditLogRepository)


### AuditLogRepository query (pseudocode)

```sql
SELECT
  ae.id, ae.action, ae.created_at, ae.metadata,
  COALESCE(su.display_name, 'System') AS actor_name,
  COALESCE(su.role, 'system')         AS actor_role
FROM audit_events ae
LEFT JOIN shop_users su ON su.id = ae.actor_user_id
WHERE
  ae.created_at >= $dateFrom
  AND ($actions::text[] IS NULL OR ae.action = ANY($actions))
ORDER BY ae.created_at DESC
LIMIT $pageSize OFFSET $offset
```

RLS (via `withTenantTx`) already enforces `shop_id` on `audit_events`. No extra WHERE clause needed.

IST midnight boundary for `today`:
```ts
const IST_OFFSET_MS = 5.5 * 60 * 60 * 1000;
const nowIST = new Date(Date.now() + IST_OFFSET_MS);
const startIST = new Date(
  Date.UTC(nowIST.getUTCFullYear(), nowIST.getUTCMonth(), nowIST.getUTCDate())
  - IST_OFFSET_MS
);
```

## 9. Story 1.7 — Logout All Devices

### API endpoint

```
POST /api/v1/auth/logout/all
```

Any authenticated role. Returns 204.

**firebase_uid source:** `req.user.uid` from `FirebaseJwtStrategy` (already on request). Follows the `/session` pattern — no TenantContext interface change, no extra DB query.

```ts
// Controller
@Post('/logout/all')
@HttpCode(204)
async logoutAll(@Req() req: Request): Promise<void> {
  const user = (req as FirebaseRequest).user;
  if (!user?.uid) throw new UnauthorizedException({ code: 'auth.missing' });
  const ctx = tenantContext.requireCurrent();
  if (!ctx.authenticated) throw new UnauthorizedException({ code: 'auth.not_authenticated' });
  const auth = ctx as AuthenticatedTenantContext;
  await this.svc.logoutAll(auth.userId, user.uid);
}

// Service
async logoutAll(userId: string, firebaseUid: string): Promise<void> {
  await this.firebase.admin().auth().revokeRefreshTokens(firebaseUid);
  await tenantContext.runWith(tenantContext.requireCurrent() as AuthenticatedTenantContext, () =>
    auditLog(this.pool, {
      action: AuditAction.AUTH_LOGOUT_ALL,
      subjectType: 'shop_user',
      subjectId: userId,
      actorUserId: userId,
      metadata: { deviceCount: 'all' },
    }),
  );
}
```

## 11. Tests

### Story 1.6

**Unit — `AuditLogRepository`**
- Returns events in `created_at DESC` order
- `pageSize > 50` is clamped to 50 before query
- Category `staff` maps to `['STAFF_INVITED','STAFF_REVOKED','STAFF_ACTIVATED']`
- No category → `$actions` is null → all events returned
- `dateRange=today` uses IST midnight boundary

**Unit — `AuthController.getAuditLog`**
- `shop_staff` role → `PolicyGuard` throws `ForbiddenException` (403)

**Integration — immutability**
- `UPDATE audit_events SET action='X' WHERE id=...` via `app_user` role → PG error: PERMISSION DENIED
- `UPDATE audit_events ...` by any role → PG error: `restrict_violation` (trigger)
- `DELETE FROM audit_events WHERE id=...` → same trigger fires

**Integration — read path**
- GET `/audit-log` with OWNER JWT → 200; `events[0].createdAt >= events[1].createdAt` (newest-first)
- GET `/audit-log?category=login` → all returned events have action in login category
- GET `/audit-log?pageSize=100` → response `pageSize` = 50 (clamped)
- GET `/audit-log` with `shop_staff` JWT → 403

**Security**
- Direct SQL `UPDATE audit_events SET action='TAMPERED' WHERE id=$id` via `app_user` pg role → PG permission denied

### Story 1.7

**Unit — `AuthService.logoutAll`**
- Calls `firebase.admin().auth().revokeRefreshTokens` with the exact `firebaseUid` passed in
- After revocation, emits `AuditAction.AUTH_LOGOUT_ALL` with `metadata.deviceCount = 'all'`

**Integration**
- After `logoutAll()`, `verifyIdToken(token, checkRevoked=true)` throws `auth/id-token-revoked` (Firebase emulator pattern)
- Re-login via `/session` issues a new valid token that passes `verifyIdToken`

**Tenant isolation**
- `logoutAll` for user in shop A → `revokeRefreshTokens` called exactly once with shop A user's UID
- No UID from shop B is passed to `revokeRefreshTokens`

## 13. Decisions Log

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Hindi label placement | Mobile-side `ACTION_HINDI` map | Follows `ROLE_HINDI` pattern; API stays language-agnostic for future EN admin console |
| firebase_uid for logoutAll | `req.user.uid` in controller | Already on request from JWT strategy; follows `/session` pattern; no TenantContext interface change |
| AuditLogRepository | New file, not merged into AuthRepository | Single-responsibility; AuthRepository is already ~100 LOC |
| Pagination style | Offset-based | Pre-revenue: max ~thousands of rows per tenant; cursor adds complexity with no benefit at this scale |
| IST date formatting | `Intl.DateTimeFormat` in mobile component | No new dep; supported in Hermes (RN); server stays UTC |
| Confirmation dialog | `Alert.alert` | Matches existing staff.tsx pattern; no BottomSheet dep added |
| SECURITY DEFINER on trigger | Yes | Ensures trigger fires even for roles that own the table; belt-and-suspenders vs REVOKE |

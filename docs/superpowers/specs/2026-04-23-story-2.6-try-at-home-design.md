# Story 2.6 — Try-at-Home Toggle Design Spec

Date: 2026-04-23  
Class: B (compressed ceremony — Codex + CI + curl smoke test)  
Complexity: XS

---

## Problem

Shopkeeper needs to toggle try-at-home availability and cap the per-booking piece count. Both values must propagate to the customer app within 60 s (NFR-P6). The `shop_settings` columns (`try_at_home_enabled`, `try_at_home_max_pieces`) already exist from migration 0006 — no schema change needed.

---

## Design Question Resolved

**FeatureFlagsCache as separate class vs extending SettingsCache with a flags method.**

**Decision: Separate class** (`packages/tenant-config/src/feature-flags.ts`), mirroring the `PermissionsCache` pattern. Reasons:
1. `SettingsCache` uses `tenantContext.requireCurrent()` internally throughout; adding flags methods with explicit `shopId` would create a mixed contract.
2. The `/feature-flags` endpoint is customer-app-facing and may be called outside the shopkeeper tenant-context path — explicit `shopId` parameter is required.
3. Clean single-responsibility; consistent with the existing `PermissionsCache` precedent.

## API Contracts

### `GET /api/v1/settings/try-at-home`

Auth: OWNER or MANAGER  
Response:
```json
{ "tryAtHomeEnabled": false, "tryAtHomeMaxPieces": 3, "etag": "\"abc123\"" }
```

### `PATCH /api/v1/settings/try-at-home`

Auth: OWNER only  
Body:
```json
{ "tryAtHomeEnabled"?: boolean, "tryAtHomeMaxPieces"?: number }
```
Validation:
- `tryAtHomeMaxPieces` must be integer 1–10 if provided
- If `tryAtHomeEnabled = false` and `tryAtHomeMaxPieces` provided → ignore piece count (save enabled=false, keep existing max)
- Defaults: enabled=false, maxPieces=3

On save:
1. Write to `shop_settings`
2. `settingsCache.invalidate()` (full invalidate — profile key; rate-lock, etc. untouched)  
   Actually: invalidate only the try-at-home slice (no separate try-at-home key in SettingsCache; service reads direct from DB for GET, no caching needed since feature-flags cache handles propagation)
3. `featureFlagsCache.invalidate(shopId)`
4. `auditLog(SETTINGS_TRY_AT_HOME_UPDATED, { before, after })`
5. Return updated values + ETag

> Note: try-at-home values are NOT cached in `SettingsCache` (they are simple scalar reads). The service reads them fresh from DB on every GET. Only the feature-flags cache (60 s TTL) serves propagation.

### `GET /api/v1/settings/feature-flags`

Auth: any authenticated role (shopkeeper + customer app stub)  
Cache: read from `FeatureFlagsCache`; on miss, load from DB and populate cache  
Response:
```json
{ "try_at_home": false, "max_pieces": 3 }
```

## FeatureFlagsCache

```
Location: packages/tenant-config/src/feature-flags.ts
Key: shop:{shopId}:feature-flags
TTL: 60 s
API:
  getFlags(shopId): Promise<{ try_at_home: boolean; max_pieces: number } | null>
  setFlags(shopId, data): Promise<void>
  invalidate(shopId): Promise<void>
```

Pattern: identical to `PermissionsCache` — explicit `shopId`, no `tenantContext` dependency.

## Mobile

### TryAtHomeToggle component

Uses `SettingsGroupCard`. Layout:
- Title: `SettingsGroupCard title` = "घर पर try करें"
- Row 1: Toggle switch, label "घर पर try करें", subtitle "Customer घर पर कुछ pieces try करके खरीद सकते हैं"
- Row 2: Piece-count stepper "अधिकतम pieces per booking" — same stepper pattern as `RateLockDurationPicker` (min 1, max 10, mountedRef guard, 1 s debounce on step)
  - Disabled + greyed when toggle is OFF

Save behavior:
- Toggle change: immediate PATCH (no debounce)
- Stepper change: 1 s debounce PATCH

Toast on success: "बदलाव सहेज लिया ✓" + medium haptic

### try-at-home.tsx screen

Same shell as `rate-lock.tsx` — `useQuery` + `useMutation`, loading spinner, renders `TryAtHomeToggle`.

### settings/index.tsx

Add nav item after `rate_lock`:
```
{ key: 'try_at_home', label: 'घर पर try करें', route: '/settings/try-at-home' }
```

## Tests

### Unit (`settings.service.spec.ts`)
- `updateTryAtHome()` toggle saves correctly (enabled true, maxPieces preserved)
- Piece count ignored when `tryAtHomeEnabled = false`
- `tryAtHomeMaxPieces = 11` throws 422
- Cache invalidated on save
- `auditLog` called with before/after

### Integration (`settings.integration.spec.ts`)
- `PATCH { tryAtHomeEnabled: true }` → `GET /feature-flags` returns `{ try_at_home: true, max_pieces: 3 }` (propagation SLA verified within 60 s TTL)

### Tenant isolation
- PATCH with tenant A JWT cannot update tenant B's `shop_settings` (verified via RLS)

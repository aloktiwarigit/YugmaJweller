# Story 3.7 — Live Stock Valuation Dashboard: Design

**Date:** 2026-04-24
**Class:** B (compressed ceremony — TDD, Codex-only review, smoke test)
**Branch:** `feat/story-3.7-valuation-dashboard`

---

## What We're Building

A live stock valuation feature that lets shopkeepers see the current market value of their
entire inventory, broken down by category, in Hindi-first UI.

---

## API Integration Adjustments (from codebase exploration)

| Spec assumption | Reality | Adaptation |
|---|---|---|
| `listProducts()` filters by status array | Only supports `status?: string` (single) | Add `listProductsForValuation()` repo method with `status IN (...)` SQL |
| `Weight.multiply(bigintRate)` | Method takes `Decimal \| number`, not `bigint` | Convert: `new Decimal(rate.perGramPaise.toString())` before multiply |
| EventEmitter2 domain events for cache invalidation | Not installed; Epic 7 TODOs in service | **TTL-only** (5-min Redis TTL). No EventEmitter2. No inventory.service.ts changes — avoids Track B coordination risk |
| Category name available on product | Only `category_id` on product row | JOIN `product_categories` in valuation SQL query |
| `ratesStale` from rates object | `CurrentRatesResult` has `fetchedAt: Date` per purity | `(now - rates.GOLD_22K.fetchedAt.getTime()) > 30 * 60 * 1000` |

## Architecture

### WS-A: InventoryValuationService

**File:** `apps/api/src/modules/inventory/inventory.valuation.service.ts`

Depends on: `InventoryRepository` (new method), `PricingService`, `INVENTORY_REDIS`

**New repository method** `listProductsForValuation(shopId: string)`:

```sql
SELECT p.id, p.metal, p.purity, p.net_weight_g, p.making_charge_override_pct,
       p.category_id, COALESCE(c.name, 'अन्य') AS category_name
FROM   products p
LEFT   JOIN product_categories c ON c.id = p.category_id
WHERE  p.status IN ('IN_STOCK', 'RESERVED', 'ON_APPROVAL', 'WITH_KARIGAR')
```

RLS enforced by `withTenantTx`. No pagination — we need all active stock.

**Valuation computation:**

1. Fetch products via `listProductsForValuation(shopId)`
2. Fetch rates via `pricingService.getCurrentRates()`
3. Check `ratesStale = (now - rates.GOLD_22K.fetchedAt) > 30 min`
4. For each product:
   - Look up rate key using `PURITY_RATE_MAP[product.purity]`
   - Unknown purity → log warning, skip (do not throw)
   - `Weight.from(product.net_weight_g).multiply(new Decimal(rate.perGramPaise.toString()))`
   - Result is paise as Decimal → `BigInt(result.toFixed(0))`
5. Group by `category_name`, sum weights (`Weight.add()`) and values (`BigInt +`)
6. Sort categories by `marketValuePaise DESC`
7. Redis cache: key `valuation:<shopId>`, TTL 300s (5 min)
   - **Cache invalidation: TTL-only.** EventEmitter2 wiring deferred to Epic 7.
   - Rationale: stock valuation freshness of 5 min is acceptable; avoids touching inventory.service.ts (Track B coordination risk) and installing @nestjs/event-emitter.

**Response types:**

```ts
export interface CategoryValuation {
  category:         string;   // Hindi category name (or 'अन्य')
  productCount:     number;
  totalWeightG:     string;   // "123.4500 g"
  marketValuePaise: bigint;
  formattedValue:   string;   // "₹12,34,567.00"
}

export interface ValuationSummary {
  categories:          CategoryValuation[];
  grandTotalPaise:     bigint;
  grandTotalFormatted: string;
  ratesFreshAt:        Date;
  ratesStale:          boolean;
  computedAt:          Date;
}
```

Formatting: `new Intl.NumberFormat('en-IN', { minimumFractionDigits: 2 }).format(Number(paise) / 100)` with `₹` prefix.

**Purity → rate key map:**

```ts
const PURITY_RATE_MAP: Record<string, keyof PurityRates> = {
  '24K': 'GOLD_24K', '22K': 'GOLD_22K', '20K': 'GOLD_20K',
  '18K': 'GOLD_18K', '14K': 'GOLD_14K',
  '999': 'SILVER_999', '925': 'SILVER_925',
};
```

## Purity → Rate key clarification

The products table stores purity values like `22K`, `24K`, `999`, `925`. The PURITY_RATE_MAP
maps these string values to `PurityRates` keys. Unmapped values (e.g. `PT950` platinum)
are skipped with a logger warning.

# WS-0.1 â€” Agent-Context Layer Design Spec

## 1. Purpose

Every Claude Code session currently reconstructs project understanding by reading
CLAUDE.md, MEMORY.md, BMAD planning artifacts, and long docs/superpowers/plans files.
This costs ~30% of each session's token budget on re-orientation before any useful
work begins. Across 44 planned gap-closure stories this is a significant waste.

This work stream creates a machine-readable context layer under `docs/agent-context/`
consisting of six compact JSON files that future sessions load instead of re-scanning
large documents. It also adds generator scripts, a validation script, and a CI gate
to keep the layer fresh.

## 2. Non-goals

- This does not generate code or modify any application source file.
- This does not create a runtime API or SDK.
- This does not replace CLAUDE.md or MEMORY.md â€” those remain authoritative; this layer
  supplements them with structured, token-efficient equivalents.
- This does not hard-block CI merges in WS-0.1 (the `docs:validate` job runs but is not
  in the `build` gate's `needs:` array; blocking is added in WS-4A).

## 3. Output files

current migration number and reservation range.

(~300 tokens).

### 3.2 `docs/agent-context/traceability.json`

Generated by `scripts/docs/extract-bmad-trace.ts`.

Primary source: YAML `implements:` arrays in every `docs/superpowers/specs/*.md` file.
These are parsed with the `yaml` package (`import { parse } from 'yaml'`).
Confirm presence with `pnpm list yaml --depth=0`; add to root `package.json` devDependencies
if absent (lightweight, no sub-dependencies).
Supplementary source: `grep -r "// FR[0-9]+"` over `apps/api/src/**/*.ts`.
Override source: `docs/agent-context/traceability-seed.json` (hand-maintained; starts
as `{ "overrides": [] }`).

Contains: one entry per FR (FR1â€“FR140) with id, title, epic, story, status, evidence
level, spec file path, primary code module, representative test file, and CI job.

Evidence levels:
- `"code-verified"` â€” FR found in spec frontmatter AND a matching test file exists.
- `"spec-only"` â€” FR in spec frontmatter but no matching test file found.
- `"missing"` â€” FR not in any spec; only in PRD. Wave assigned from the architecture plan.

Schema per entry:

```json

  "epic": "E1",
  "story": "1.1",

  "ciJob": "integration"

```

A `"gaps"` top-level array lists all `status: "missing"` entries for quick scanning.

Estimated token cost: ~1,200 tokens for the full file (140 FR entries Ă— ~9 tokens each).

### 3.3 `docs/agent-context/decision-index.json`

Generated by `scripts/docs/extract-adrs.ts`.

Parses every `docs/adr/*.md` file. ADRs have no YAML frontmatter â€” H1 title, inline-bold
metadata lines (`**Status:**`, `**Date:**`), and H2 sections (`## Context`,
`## Decision`, `## Consequences`, `## Alternatives Considered`). The extractor uses
regex to pull: ADR ID from filename, title from H1, status from bold `Status:` line,
`supersedes` / `supersededBy` from status prose, and decision bullet points from under
the `## Decision` H2.

Schema per entry:

```json
{
  "id": "ADR-0015",
  "title": "Firebase Auth for Phone OTP",
  "status": "accepted | superseded | draft",
  "supersedes": "ADR-0001",
  "supersededBy": null,
  "rules": [
    "Firebase Auth handles phone OTP end-to-end",
    "MSG91 deferred unless Firebase Auth cannot fit a specific flow"
  ],
  "path": "docs/adr/0015-firebase-auth.md"
}
```

Superseded ADRs are included with `"status": "superseded"` so agents know to ignore
their rules. Agents reading decision-index.json must not act on rules from superseded
entries.

Estimated token cost: ~600 tokens (16 ADRs Ă— ~38 tokens each).

Generated by `scripts/docs/scan-docs.ts`.

- `C:\Users\alokt\.claude\projects\â€¦\memory\*.md` (session memory â€” loaded via MEMORY.md)

### 3.6 `docs/agent-context/acceptance-evidence.json`

Generated by `scripts/docs/extract-acceptance.ts`.

Gap-closure stories (WS-0.1 â†’ WS-4C, 44 stories) are pre-populated from

## 4. Generator scripts

All scripts live in `scripts/docs/`. Runtime: `tsx` (already in root devDependencies).
Each script is self-contained, reads from project root, writes to `docs/agent-context/`.

### 4.1 `scan-repo-context.ts`

Reads:
- Root `package.json` â†’ workspaces, devDependencies versions
- `turbo.json` â†’ pipeline task names
- `apps/api/src/modules/` â†’ `fs.readdirSync` to list module names
- `packages/db/src/migrations/` â†’ last migration filename to extract sequence number
- CLAUDE.md Â§ invariants section â†’ static parse (known section markers)

Produces: `project.context.json`.

Key implementation note: CLAUDE.md invariants section starts at the `### Data model`
heading and runs through `### Compliance enforcement`. The script extracts bullet items
from those two sub-sections using line-level regex rather than full Markdown parsing.
The section is stable and unlikely to shift without a deliberate CLAUDE.md edit.

### 4.2 `extract-adrs.ts`

Reads: all `.md` files in `docs/adr/` sorted by filename.

Parsing approach per file:
1. Extract ADR ID from filename prefix (`0015` â†’ `ADR-0015`).
2. Extract title from first H1 (`# 0015 â€” ...`).
3. Extract `Status:` from inline bold line using regex `/\*\*Status:\*\*\s*(.+)/`.
4. Parse `supersedes`/`supersededBy` from status prose if present
   (pattern: `supersedes ADR-NNNN`).
5. Extract decision bullets from lines between `## Decision` and next `##` heading.
6. Rules array = decision bullet items, stripped of leading `-` or `*`.

No external Markdown parser needed. Produces: `decision-index.json`.

### 4.3 `extract-bmad-trace.ts`

Step 1 â€” Parse spec frontmatter:
- Walk `docs/superpowers/specs/*.md`.
- Parse YAML frontmatter with `import { parse } from 'yaml'` (yaml@^2 is a transitive
  dep via several BMAD packages; confirm presence; if absent, add to root devDependencies).
- Extract: `implements` (array of FR IDs like `["FR8", "FR9"]`), `adrs`, `nfrs`,
  story ID from filename (`2026-04-18-story-1.1-â€¦` â†’ `"1.1"`).

Step 2 â€” Grep code:
- Run `child_process.execSync('grep -rh "// FR[0-9]+" apps/api/src', {encoding:'utf8'})`.
- Parse matches to build a map of `{ "FR8": ["apps/api/src/modules/auth/auth.service.ts"] }`.
- Windows: use `grep` via git-bash; fall back to manual file walk + regex if grep
  unavailable (detect with `process.platform === 'win32'`; use `fast-glob` + `fs.readFile`
  in that case).

Step 3 â€” Load overrides:
- Read `docs/agent-context/traceability-seed.json` (`{ "overrides": [] }`).
- Merge: seed overrides take precedence over auto-derived entries.

Step 4 â€” Build FR entries for FR1â€“FR126 (from `_bmad-output/planning-artifacts/prd.md`
FR list) plus FR127â€“FR140 (from `docs/prd-addendum-customer-storefront.md`).
- Match each FR to a spec's `implements` array.
- Assign `evidence` level per rules in Â§3.2.

Produces: `traceability.json`.

### 4.4 `scan-docs.ts`

Walks `docs/**/*.md` and `_bmad-output/**/*.md` using `fast-glob`.
For each file: `fs.statSync` for size, estimate tokens as `Math.ceil(lines * 10)`.
Applies `avoidByDefault: true` to paths matching known patterns (see Â§3.5).
Infers `kind` from directory segment (e.g. `docs/adr/` â†’ `"adr"`,
`docs/superpowers/specs/` â†’ `"spec"`).

Produces: `doc-index.json`.

### 4.5 `extract-acceptance.ts`

Walk `docs/superpowers/specs/*.md`, parse frontmatter (same yaml parser as 4.3).
Extract story ID from filename. Build initial entry with `frs`, `class`, spec path.

Grep test dirs for story ID fragments:
```
apps/api/test/**/*.ts
apps/shopkeeper/src/**/__tests__/**/*.ts  (if exists)
packages/*/src/**/*.spec.ts
```

Match heuristic (two-pass):
1. Primary: test filename or path contains the story ID string (e.g. `"1.1"` â†’ match
   any file with `*1.1*` in its name or path).
2. Secondary: look up the primary `codeModule` for the story's FRs from
   `traceability.json`, then find test files under `apps/api/test/` whose name
   matches the module name (e.g. story 1.1 â†’ `codeModule: "auth"` â†’ match
   `*auth*.test.ts`).

Load gap-closure story stubs from `traceability-seed.json` overrides where
`"status": "planned"`.

Produces: `acceptance-evidence.json`.

### 4.6 `validate-agent-context.ts`

Orchestrator script. For each of the 5 generator scripts:
1. Call the generator function (scripts export a `generate(): Promise<object>` function
   in addition to a CLI entry point that calls `generate()` and writes the file).
2. Compare the generated object against the committed JSON file using deep-diff.
3. Compute drift as: `changedEntries / totalEntries`.

Exit conditions:
- `drift === 0` for all files: exit 0, print "docs/agent-context/ is up to date".
- `0 < drift â‰¤ 0.05` for any file: exit 0, print warning "Drift detected in
  {file}: {n} entries changed. Run pnpm docs:context to update."
- `drift > 0.05` for any file: exit 1, print "Significant drift in {file}: run
  pnpm docs:context and commit before merging."
- Referenced file path in `task-routing.json` does not exist on disk: exit 1.
- Referenced `specFile` path in `acceptance-evidence.json` does not exist: exit 1.

Rationale for 5% threshold: a single new story landing adds ~1â€“2 entries out of 140+
total, which is <2%. The 5% gate prevents false-positive failures on small incremental
additions while catching bulk drift (e.g., spec files renamed, ADR IDs changed).

### 5.2 `.github/workflows/ship.yml` â€” new job

```yaml

  name: docs-validate
  runs-on: ubuntu-latest
  needs: install
  steps:
    - uses: actions/checkout@v4
    - uses: pnpm/action-setup@v3
      with: { version: 9 }
    - uses: actions/setup-node@v4
      with: { node-version: '20', cache: 'pnpm' }
    - run: pnpm install --frozen-lockfile
    - run: pnpm docs:validate
```

**Blocking behaviour**: `docs-validate` is NOT added to the `build` gate job's
`needs:` array in WS-0.1. It runs on every PR and reports drift but does not block
merge. This avoids false-positive breakage during Phase 1â€“3 as new stories land and
traceability entries are added. Blocking is enabled in WS-4A when coverage â‰Ą95%.

### 5.3 `turbo.json` â€” no changes

`docs:context` and `docs:validate` operate across the whole repo and are not
per-workspace tasks. They follow the existing pattern of `semgrep` and `db:reset`
(root scripts only).

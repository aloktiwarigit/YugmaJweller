## 1. Context and scope

Story 6.8 (April 2026, migration `0036_customer_dpdpa_deletion.sql`) shipped owner-initiated DPDPA deletion through the CRM controller. It scrubs PII immediately, schedules a 30-day hard-delete via BullMQ + a daily safety-net sweep, and detaches invoices. Story 19.7 closes the customer-self-initiated half:

1. Customer triggers their own deletion from the customer app.
2. Cascade extends to the four tables added since 6.8 that the existing pipeline does NOT touch: `wishlists`, `product_reviews`, `rate_lock_bookings`, `try_at_home_bookings`.
3. Customer-web ships a delete-account page (none today).
4. Customer-mobile's existing 2-step inline button on the Profile tab is replaced by a dedicated AC-compliant screen with reason picker and Hindi type-to-confirm phrase.
5. Migration 0075 closes the schema gaps that block hard-delete today.

This is a **brownfield enhancement**, not greenfield. The endpoint `DELETE /api/v1/crm/customer/me`, the mobile axios call `customerSelfDelete()`, and a 2-step confirm UI already exist on main. The actual delta is (a) extending the existing pipeline, (b) a new web UI, (c) enhancing mobile UX to AC spec, (d) the small migration for schema gaps.

## 3. Locked design decisions

### 3.1 Reuse existing pipeline (not parallel)

`DpdpaDeletionService.requestDeletion(ctx, customerId, requestedBy, reason?)` is extended; no new "deleteSelf" code path. Customer-self vs owner-initiated differ only in `requestedBy` value (already `'customer' | 'owner'`) and the optional `reason` field (new).

### 3.2 Reviews anonymisation: NULL `customer_id` + `LEFT JOIN COALESCE`

- Migration 0075: `product_reviews.customer_id` → `NULLABLE`; FK changes to `ON DELETE SET NULL`.
- Soft-delete (day 0): `UPDATE product_reviews SET customer_id = NULL WHERE customer_id = $1 AND shop_id = current_setting('app.current_shop_id')::uuid`.
- `reviews.repository.ts::listByProduct` changes from `INNER JOIN customers` to `LEFT JOIN customers` with `COALESCE(split_part(c.name, ' ', 1), 'एक ग्राहक') AS customer_first_name`.
- Hard-delete (day 30): nothing additional needed — FK `ON DELETE SET NULL` keeps the review with `customer_id IS NULL`.

### 3.3 Try-at-home DISPATCHED state BLOCKS deletion

Mirrors the existing `crm.deletion.open_invoices` pattern. If `try_at_home_bookings.status = 'DISPATCHED'` for this customer at deletion time, `softDeleteAtomic` raises `UnprocessableEntityException { code: 'crm.deletion.try_at_home_in_flight' }`. The mobile + web UIs surface "घर पर ट्राय का सामान अभी आपके पास है — पहले लौटाएँ".

REQUESTED bookings transition to `'CANCELLED'` (added to enum in 0075). RETURNED / CONVERTED_TO_SALE / EXPIRED bookings are left untouched (already settled).

### 3.4 Rate-lock cancellation + manual refund

Soft-delete sets `status = 'CANCELLED'` for any booking where `customer_id = $1` AND `status IN ('PENDING_PAYMENT', 'ACTIVE')`. The `'CANCELLED'` state already exists in the enum (verified at `0045_rate_lock_bookings.sql:19`).

If a cancelled booking has `deposit_paid_paise > 0`, the shopkeeper must process the refund manually. An audit log entry (`CRM_CUSTOMER_SOFT_DELETED` with `after.rateLockRefundsPending`) flags the operator. Automatic refund is OUT of scope — the payments adapter has no refund API plumbed today.

### 3.5 UX: reason picker + Hindi type-to-confirm on both apps

#### Mobile (new screen at `apps/customer-mobile/app/profile/delete-account.tsx`)

- Reached via "डेटा हटाएँ" link from `(tabs)/profile.tsx` (replacing the existing inline 3-step button).
- Heading: "क्या आप वाक़ई अपना खाता हटाना चाहते हैं?"
- Body: explanation of irreversibility, 30-day shell window, what happens to past invoices (kept) vs PII (scrubbed now).
- Radio group (4 options):
  - `no-need` — "मुझे ज़रूरत नहीं"
  - `privacy` — "गोपनीयता की चिंता"
  - `other-jeweller` — "दूसरे जौहरी से खरीद रहा"
  - `other` — "अन्य" (reveals optional 200-char free-text input)
- Type-to-confirm input: must equal exactly `मेरा डेटा मिटाएँ` (no normalisation; literal Hindi string).
- Submit button only enabled when BOTH reason selected AND phrase matches.
- On submit → `customerSelfDelete({ reason, reasonText })` → on success: clear session + `auth().signOut()` + navigate to a success screen ("आपका खाता मिटा दिया गया है").
- Hindi error mapping (typed-error → user copy) for: `crm.deletion.open_invoices`, `crm.deletion.already_requested`, `crm.deletion.try_at_home_in_flight`.

#### Web (new page at `apps/customer-web/app/profile/delete-account/page.tsx`)

customer-web has no broader client-side Firebase auth wired today (no `/sign-in` route, no customer-session cookie, no shared auth context). The delete-account page handles its own auth scope:

- A thin SSR page wrapper resolves the tenant `shopId` via `/api/v1/tenant/boot?slug=<NEXT_PUBLIC_SHOP_SLUG>` and passes it into a client component.
- The client component runs a self-contained Firebase phone-OTP gate first (using `firebase` v11 web SDK, already in deps; invisible reCAPTCHA). After OTP confirm, switches stage to the deletion form.
- The deletion form (same UX as mobile: 4-option radio + Hindi type-to-confirm "मेरा डेटा मिटाएँ") submits via client-side `fetch` carrying `Authorization: Bearer <Firebase ID token>` + `X-Tenant-Id: <resolved shopId>`. No server actions, no session cookies.
- Firebase Auth helpers (`signInWithPhoneNumber`, `RecaptchaVerifier`, `getAuth`) are imported through the per-app wrapper at `apps/customer-web/src/auth/firebase-customer.ts`, not directly from `firebase/auth`, to satisfy the `ops/semgrep/no-firebase-client-outside-auth-client.yaml` rule. The wrapper directory is added to the rule's `exclude` list (mirroring the existing `apps/shopkeeper/src/auth/**` exception).
- All `NEXT_PUBLIC_*` env reads in the client bundle use dot-property access (`process.env.NEXT_PUBLIC_API_BASE`) — Next.js only inlines dot-access reads at build time; bracket access returns `undefined` in the browser.
- On 202 success: client-side `auth.signOut()` + `router.push('/profile/delete-account/done')`.
- Accessibility: `role="radiogroup"`, `aria-required`, focus-visible, keyboard navigable; type-to-confirm input has `aria-describedby` linking to the phrase rendered above it.

### 3.6 Endpoint accepts reason in body

`DELETE /api/v1/crm/customer/me` accepts optional body validated by Zod:

```typescript
const DeletionBodySchema = z.object({
  reason: z.enum(['no-need','privacy','other-jeweller','other']).optional(),
  reasonText: z.string().max(200).optional(),
}).strict().default({});
```

The `.default({})` is load-bearing: NestJS runs `ZodValidationPipe` on raw `undefined` when the request body is absent, BEFORE any TypeScript default parameter fires. Without `.default({})` the schema rejects `undefined` and the existing no-body customer-mobile callers regress to 400. Verified by an integration test that sends a DELETE with no body and expects 202.

Empty body remains valid (existing behavior preserved). `reason` persists to `customers.deletion_reason` column added in 0075; `reasonText` persists to `customers.deletion_reason_text` and is NEVER copied to the audit log.

### 3.7 Audit events stay as existing enum values

`CRM_CUSTOMER_DELETION_REQUESTED` + `CRM_CUSTOMER_SOFT_DELETED` fire as today. `after.requestedBy = 'customer'` (existing). New: `after.reason` (structured enum value only) + `after.rateLockRefundsPending: number` (count of active rate-lock bookings cancelled with paid deposit) + `after.cascadeCounts: { wishlists, reviews, rateLocks, tryAtHome }` (sanity diagnostics).

**`reasonText` is NOT copied into the audit payload** — free customer text may contain a phone, PAN, address, or email the customer pasted in. It's persisted only on the `customers.deletion_reason_text` column (customer's own row). This guard is verified by an explicit integration assertion (no raw phone or PAN matched by regex in any audit row).

No raw phone or PAN in any audit payload — guaranteed by existing 6.8 code + the reasonText exclusion above. Verified by integration test (§7).

### 3.8 No re-OTP gate

Type-to-confirm phrase is the explicit-intent gate. The customer is already past CustomerAuthGuard (valid Firebase ID token within the past hour). No fresh OTP requirement — matches Apple/Google account-delete UX.

### 3.9 Firebase Auth user record persists

After our DB cleanup the Firebase Auth record (UID + phone) is left intact. The next OTP attempt by the same phone will succeed at Firebase but fail at our `customer-auth.guard.ts` (line 78: `WHERE phone = $1 AND shop_id = $2 AND deleted_at IS NULL`). Result: `customer.not_found` → 401. The shopkeeper re-adds the customer through their CRM. The phone-hash trick in `softDeleteAtomic` keeps the unique-index slot free for the new row.

Trade-off accepted: customer cannot self-re-register. Consistent with the platform's multi-tenant model (customers are shopkeeper-owned). Self-provision is a follow-up story.

### 3.10 Migration 0075 contents

```sql
-- 0075_customer_self_deletion_extensions.sql
--
-- Story 19.7 — extends 0036 DPDPA deletion to four tables added after 6.8 shipped:
--   wishlists, product_reviews, rate_lock_bookings, try_at_home_bookings.
--
-- The hard-delete in 0036's worker currently fails FK constraints whenever
-- a customer has any of these rows. This migration unblocks it AND adds the
-- columns needed for the customer-self reason picker shipping with 19.7.

BEGIN;

-- 1. customers.deletion_reason — customer-supplied reason captured at soft-delete.
ALTER TABLE customers ADD COLUMN deletion_reason TEXT
  CHECK (deletion_reason IS NULL OR deletion_reason IN
    ('no-need','privacy','other-jeweller','other'));

ALTER TABLE customers ADD COLUMN deletion_reason_text TEXT
  CHECK (deletion_reason_text IS NULL OR length(deletion_reason_text) <= 200);

-- 2. product_reviews: customer_id NULLABLE + FK SET NULL on hard-delete.
ALTER TABLE product_reviews ALTER COLUMN customer_id DROP NOT NULL;
ALTER TABLE product_reviews DROP CONSTRAINT product_reviews_customer_id_fkey;
ALTER TABLE product_reviews ADD CONSTRAINT product_reviews_customer_id_fkey
  FOREIGN KEY (customer_id) REFERENCES customers(id) ON DELETE SET NULL;

-- 3. try_at_home_bookings: add 'CANCELLED' status for customer-deletion cascade.
ALTER TABLE try_at_home_bookings DROP CONSTRAINT try_at_home_bookings_status_check;
ALTER TABLE try_at_home_bookings ADD CONSTRAINT try_at_home_bookings_status_check
  CHECK (status IN ('REQUESTED','DISPATCHED','RETURNED','CONVERTED_TO_SALE','EXPIRED','CANCELLED'));

-- 4. rate_lock_bookings: detach customer_id at hard-delete time.
--    Without this, hardDeleteAtomic FK-violates at day 30 whenever a customer
--    has any rate-lock row (cancelled or terminal). The booking row stays for
--    the shopkeeper's audit trail; only the customer link is severed when the
--    customer row is removed.
ALTER TABLE rate_lock_bookings ALTER COLUMN customer_id DROP NOT NULL;
ALTER TABLE rate_lock_bookings DROP CONSTRAINT rate_lock_bookings_customer_id_fkey;
ALTER TABLE rate_lock_bookings ADD CONSTRAINT rate_lock_bookings_customer_id_fkey
  FOREIGN KEY (customer_id) REFERENCES customers(id) ON DELETE SET NULL;

-- 5. try_at_home_bookings: same detach behavior on customer_id.
ALTER TABLE try_at_home_bookings ALTER COLUMN customer_id DROP NOT NULL;
ALTER TABLE try_at_home_bookings DROP CONSTRAINT try_at_home_bookings_customer_id_fkey;
ALTER TABLE try_at_home_bookings ADD CONSTRAINT try_at_home_bookings_customer_id_fkey
  FOREIGN KEY (customer_id) REFERENCES customers(id) ON DELETE SET NULL;

COMMIT;
```

Why no other migration entries:

- `wishlists`: no schema change needed. Soft-delete does an explicit `DELETE FROM wishlists WHERE customer_id=...`; the table already has DELETE granted to app_user (`0047_reviews_wishlist.sql:65`). `wishlists.customer_id` does not need ON DELETE SET NULL because the row is hard-deleted at day 0 inside `softDeleteAtomic`.
- `rate_lock_bookings.status`: `'CANCELLED'` is already in the status enum (`0045_rate_lock_bookings.sql:19`).
- `product_reviews` UPDATE grant: already covered by `0059_reviews_wishlist_update_grant.sql` (`GRANT UPDATE ON product_reviews TO app_user`). The anonymisation UPDATE inside `softDeleteAtomic` runs as `app_user` and succeeds.
- RLS policies on all four tables already filter on `shop_id`; the cascade runs inside `withTenantTx` so RLS continues to scope correctly.

**Drizzle schema updates accompanying this migration** (must land in the same commit as the SQL):

- `packages/db/src/schema/customers.ts` — add `deletionReason: text('deletion_reason')` and `deletionReasonText: text('deletion_reason_text')`. Without these, `@goldsmith/db` consumers cannot read or write the new columns through Drizzle.
- `packages/db/src/schema/product-reviews.ts` — change `customer_id: uuid('customer_id').notNull().references(() => customers.id)` to `customer_id: uuid('customer_id').references(() => customers.id, { onDelete: 'set null' })`. The `.notNull()` is removed; the FK action mirrors the SQL.

Both edits prevent type/runtime drift between the migrated DB and the TypeScript schema exports.

## 7. Runtime smoke plan (non-negotiable for Class A)

**Customer-mobile** (Pixel_6_Pravesh emulator per CLAUDE.md Maestro setup):

1. `EXPO_PUBLIC_DEV_AUTH=1` + dev API on `3001` + Metro on `8081`.
2. Sign in via dev-mock → navigate to Profile tab.
3. Tap "डेटा हटाएँ" → new screen loads.
4. Pick reason "गोपनीयता की चिंता".
5. Type-to-confirm input: paste "मेरा डेटा मिटाएँ" character-by-character; confirm submit becomes enabled.
6. Submit → success screen.
7. Verify cleared session + cannot navigate back into authed routes.
8. Run `adb screencap` at each step; attach to merge-train notes.

**Customer-web** (Next.js dev server on `3000`):

1. `npm run dev` from `apps/customer-web/`.
2. Sign in via Firebase emulator (or use a seed authed cookie) → navigate to `/profile`.
3. Click "खाता हटाएँ" link → land on `/profile/delete-account`.
4. Same submit flow; verify SSR redirect when revisiting `/profile/delete-account` after sign-out.

**Audit-log query** (against the dev DB after both flows):

```sql
SELECT action, subject_id, actor_user_id, after, created_at
FROM audit_log
WHERE action IN ('CRM_CUSTOMER_DELETION_REQUESTED','CRM_CUSTOMER_SOFT_DELETED')
ORDER BY created_at DESC LIMIT 4;
```

Verify: 2 rows per deletion (one each), `after` JSON contains `requestedBy='customer'`, `reason`, `cascadeCounts`. No raw phone or PAN anywhere.

## 8. Out of scope (deferred)

1. **Customer-self provisioning** — no API to create a `customers` row from customer-side. Required for full AC #12 closure. Separate follow-up story (P1).
2. **Automatic refund on rate-lock CANCELLED with paid deposit** — payments adapter has no refund API plumbed. Manual shopkeeper process. Separate story.
3. **Firebase Auth user deletion** — Admin SDK call deferred; preserves customer's Firebase identity for potential re-registration once self-provision lands.
4. **Account anonymisation as alternative to deletion** — explicitly out per epic.
5. **Data export (Right to Portability)** — separate FR, separate story.
6. **Admin-initiated customer deletion via customer-web** — existing 6.8 owner path handles this from shopkeeper-mobile.

## 9. Known gaps in main caught by this design

1. **Latent bug:** `hardDeleteAtomic` on main fails FK on any customer that has wishlist/review/rate-lock/try-at-home rows. Story 6.8 wasn't tested with these later-arriving tables. 19.7 fixes this; the hard-delete worker becomes correct.
2. **Audit gap:** existing pipeline doesn't record cascade counts, making post-deletion auditing imprecise. 19.7 adds `cascadeCounts` to the soft-delete event.
3. **UX gap:** existing mobile inline 3-step button doesn't capture reason or require type-to-confirm — below the AC bar for "irreversible" actions. 19.7 brings it to AC.

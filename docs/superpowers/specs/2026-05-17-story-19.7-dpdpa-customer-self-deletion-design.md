## 1. Context and scope

Story 6.8 (April 2026, migration `0036_customer_dpdpa_deletion.sql`) shipped owner-initiated DPDPA deletion through the CRM controller. It scrubs PII immediately, schedules a 30-day hard-delete via BullMQ + a daily safety-net sweep, and detaches invoices. Story 19.7 closes the customer-self-initiated half:

1. Customer triggers their own deletion from the customer app.
2. Cascade extends to the four tables added since 6.8 that the existing pipeline does NOT touch: `wishlists`, `product_reviews`, `rate_lock_bookings`, `try_at_home_bookings`.
3. Customer-web ships a delete-account page (none today).
4. Customer-mobile's existing 2-step inline button on the Profile tab is replaced by a dedicated AC-compliant screen with reason picker and Hindi type-to-confirm phrase.
5. Migration 0075 closes the schema gaps that block hard-delete today.

This is a **brownfield enhancement**, not greenfield. The endpoint `DELETE /api/v1/crm/customer/me`, the mobile axios call `customerSelfDelete()`, and a 2-step confirm UI already exist on main. The actual delta is (a) extending the existing pipeline, (b) a new web UI, (c) enhancing mobile UX to AC spec, (d) the small migration for schema gaps.

## 3. Locked design decisions

### 3.1 Reuse existing pipeline (not parallel)

`DpdpaDeletionService.requestDeletion(ctx, customerId, requestedBy, reason?)` is extended; no new "deleteSelf" code path. Customer-self vs owner-initiated differ only in `requestedBy` value (already `'customer' | 'owner'`) and the optional `reason` field (new).

### 3.2 Reviews anonymisation: NULL `customer_id` + `LEFT JOIN COALESCE`

- Migration 0075: `product_reviews.customer_id` ‚Üí `NULLABLE`; FK changes to `ON DELETE SET NULL`.
- Soft-delete (day 0): `UPDATE product_reviews SET customer_id = NULL WHERE customer_id = $1 AND shop_id = current_setting('app.current_shop_id')::uuid`.
- `reviews.repository.ts::listByProduct` changes from `INNER JOIN customers` to `LEFT JOIN customers` with `COALESCE(split_part(c.name, ' ', 1), 'ý§èý§ï ý§óý•çý§∞ý§æý§πý§ï') AS customer_first_name`.
- Hard-delete (day 30): nothing additional needed ‚Äî FK `ON DELETE SET NULL` keeps the review with `customer_id IS NULL`.

### 3.3 Try-at-home DISPATCHED state BLOCKS deletion

Mirrors the existing `crm.deletion.open_invoices` pattern. If `try_at_home_bookings.status = 'DISPATCHED'` for this customer at deletion time, `softDeleteAtomic` raises `UnprocessableEntityException { code: 'crm.deletion.try_at_home_in_flight' }`. The mobile + web UIs surface "ý§òý§∞ ý§™ý§∞ ý§üý•çý§∞ý§æý§Ø ý§ïý§æ ý§∏ý§æý§Æý§æý§® ý§Öý§≠ý•Ä ý§Üý§™ý§ïý•á ý§™ý§æý§∏ ý§πý•à ‚Äî ý§™ý§πý§≤ý•á ý§≤ý•åý§üý§æý§èý§Å".

REQUESTED bookings transition to `'CANCELLED'` (added to enum in 0075). RETURNED / CONVERTED_TO_SALE / EXPIRED bookings are left untouched (already settled).

### 3.4 Rate-lock cancellation + manual refund

Soft-delete sets `status = 'CANCELLED'` for any booking where `customer_id = $1` AND `status IN ('PENDING_PAYMENT', 'ACTIVE')`. The `'CANCELLED'` state already exists in the enum (verified at `0045_rate_lock_bookings.sql:19`).

If a cancelled booking has `deposit_paid_paise > 0`, the shopkeeper must process the refund manually. An audit log entry (`CRM_CUSTOMER_SOFT_DELETED` with `after.rateLockRefundsPending`) flags the operator. Automatic refund is OUT of scope ‚Äî the payments adapter has no refund API plumbed today.

### 3.6 Endpoint accepts reason in body

`DELETE /api/v1/crm/customer/me` accepts optional body validated by Zod:

```typescript
const DeletionBodySchema = z.object({
  reason: z.enum(['no-need','privacy','other-jeweller','other']).optional(),
  reasonText: z.string().max(200).optional(),
}).strict();
```

Empty body remains valid (existing behavior preserved). `reason` persists to `customers.deletion_reason` column added in 0075.

### 3.7 Audit events stay as existing enum values

`CRM_CUSTOMER_DELETION_REQUESTED` + `CRM_CUSTOMER_SOFT_DELETED` fire as today. `after.requestedBy = 'customer'` (existing). New: `after.reason` + `after.reasonText` (if provided) + `after.rateLockRefundsPending: number` (count of active rate-lock bookings cancelled with paid deposit) + `after.cascadeCounts: { wishlists, reviews, rateLocks, tryAtHome }` (sanity diagnostics).

No raw phone or PAN in any audit payload ‚Äî already guaranteed by existing code. Verified by integration test (¬ß7).

### 3.8 No re-OTP gate

Type-to-confirm phrase is the explicit-intent gate. The customer is already past CustomerAuthGuard (valid Firebase ID token within the past hour). No fresh OTP requirement ‚Äî matches Apple/Google account-delete UX.

### 3.9 Firebase Auth user record persists

After our DB cleanup the Firebase Auth record (UID + phone) is left intact. The next OTP attempt by the same phone will succeed at Firebase but fail at our `customer-auth.guard.ts` (line 78: `WHERE phone = $1 AND shop_id = $2 AND deleted_at IS NULL`). Result: `customer.not_found` ‚Üí 401. The shopkeeper re-adds the customer through their CRM. The phone-hash trick in `softDeleteAtomic` keeps the unique-index slot free for the new row.

Trade-off accepted: customer cannot self-re-register. Consistent with the platform's multi-tenant model (customers are shopkeeper-owned). Self-provision is a follow-up story.

### 3.10 Migration 0075 contents

```sql
-- 0075_customer_self_deletion_extensions.sql
--
-- Story 19.7 ‚Äî extends 0036 DPDPA deletion to four tables added after 6.8 shipped:
--   wishlists, product_reviews, rate_lock_bookings, try_at_home_bookings.
--
-- The hard-delete in 0036's worker currently fails FK constraints whenever
-- a customer has any of these rows. This migration unblocks it AND adds the
-- columns needed for the customer-self reason picker shipping with 19.7.

BEGIN;

-- 1. customers.deletion_reason ‚Äî customer-supplied reason captured at soft-delete.
ALTER TABLE customers ADD COLUMN deletion_reason TEXT
  CHECK (deletion_reason IS NULL OR deletion_reason IN
    ('no-need','privacy','other-jeweller','other'));

ALTER TABLE customers ADD COLUMN deletion_reason_text TEXT
  CHECK (deletion_reason_text IS NULL OR length(deletion_reason_text) <= 200);

-- 2. product_reviews: customer_id NULLABLE + FK SET NULL on hard-delete.
ALTER TABLE product_reviews ALTER COLUMN customer_id DROP NOT NULL;
ALTER TABLE product_reviews DROP CONSTRAINT product_reviews_customer_id_fkey;
ALTER TABLE product_reviews ADD CONSTRAINT product_reviews_customer_id_fkey
  FOREIGN KEY (customer_id) REFERENCES customers(id) ON DELETE SET NULL;

-- 3. try_at_home_bookings: add 'CANCELLED' status for customer-deletion cascade.
ALTER TABLE try_at_home_bookings DROP CONSTRAINT try_at_home_bookings_status_check;
ALTER TABLE try_at_home_bookings ADD CONSTRAINT try_at_home_bookings_status_check
  CHECK (status IN ('REQUESTED','DISPATCHED','RETURNED','CONVERTED_TO_SALE','EXPIRED','CANCELLED'));

COMMIT;
```

Why no other migration entries:

- `wishlists`: no schema change needed. Soft-delete does an explicit `DELETE FROM wishlists WHERE customer_id=...`; the table already has DELETE granted to app_user (`0047_reviews_wishlist.sql:65`).
- `rate_lock_bookings`: `'CANCELLED'` is already in the status enum (`0045_rate_lock_bookings.sql:19`).
- RLS policies on all four tables already filter on `shop_id`; the cascade runs inside `withTenantTx` so RLS continues to scope correctly.
